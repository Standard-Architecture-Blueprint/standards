# 2. ADR (Architecture Decision Records)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) プロジェクトにおける重要な技術的決定の記録（ADR）を管理する。AP試験の「システム方式設計」における決定プロセスを明文化し、将来の保守者や監査担当者が「設計の意図（Design Intent）」を正確に把握できるようにすることを目的とする。

## 🏛️ 管理規約 (Governance Rules)

### 1. 記録すべき内容
以下の項目に該当する決定が行われた場合、必ず ADR を作成しなければならない。
- **技術スタックの選定**: プログラミング言語、フレームワーク、DB、ミドルウェアの採用。
- **アーキテクチャ・パターン**: 通信プロトコル（REST/gRPC）、非同期処理の有無、データの整合性モデル。
- **重要な設計方針**: 共通ライブラリの分離ルール、認証・認可の方式。

### 2. 記述形式 (Template Standard)
デファクトスタンダードである **MADR (Markdown Architecture Decision Records)** 形式をベースとする。

- **Context (背景)**: 解決すべき課題、制約事項（時間、予算、技術的限界）。
- **Decision (決定)**: 採用した解決策。
- **Consequences (結果・影響)**: 決定によるメリット（正の影響）と、許容したデメリット（負の影響/トレードオフ）。

### 3. 命名規則
ファイル名は `[4桁の連番]-[簡潔なタイトル].md` とする。
（例：`0001-use-java21-spring-boot-3.md`）

## ⚖️ アーキテクチャ決定の原則 (Architectural Principles)
AP試験の「システムアーキテクチャ」の評価指標に基づき、以下の観点で意思決定を評価・記録する。

1.  **経済性（コスト）**: 開発工数、ランニングコスト、学習コストの妥当性。
2.  **性能・信頼性**: スループット、可用性、耐障害性への影響。
3.  **保守性・拡張性**: 将来の技術変更や機能追加の容易さ。
4.  **相互運用性**: 13のリポジトリ間、および外部システムとの連携のしやすさ。

## 📝 プロセス
1.  **提案**: `templates/adr/template.md` を使用して下書きを作成し、プルリクエストを出す。
2.  **審議**: 開発メンバーおよびアーキテクトによるレビューを行い、論理的整合性を検証する。
3.  **承認**: 合意形成後、`Accepted` 状態としてマージする。決定が覆った場合は、削除せず `Superseded`（代替済み）として記録を残す。
