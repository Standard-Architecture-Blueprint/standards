# ADR 0004: データ永続化戦略（RDBMS と インフラ独立性）

- **Status**: Accepted
- **Date**: 2026-05-03
- **Deciders**: System Architect

## 1. Context (背景)
エンタープライズシステムにおいて、データの整合性（ACID特性）は最も重要な要素である。
また、特定のクラウドベンダーやデータベース製品に依存しすぎる設計（ベンダーロックイン）を避け、標準的な SQL や ORM プロトコルを用いることで、将来的なインフラ移行や技術スタックの刷新に対応できる「汎用的なデータアクセス構造」を構築する必要がある。

## 2. Decision (決定)
データの永続化に関して、以下の戦略を採用する。

- **Primary Storage**: **RDBMS (Relational Database Management System)**
  - 理由: 複雑なデータリレーションの管理と、トランザクションによる強力なデータ整合性を保証するため。開発・検証環境では PostgreSQL 等のオープンソース製品を推奨する。
- **Object-Relational Mapping (ORM)**: **Spring Data JPA (Hibernate)**
  - 理由: Java オブジェクトとテーブル構造を疎結合に保つため。Repository パターンを採用し、ビジネスロジックが特定の SQL 命令に依存しないように設計する。
- **Migration Tool**: **Flyway** または **Liquibase**
  - 理由: データベースのスキーマ変更をコードとして管理（Versioned Migration）し、環境間での差異を失くすため。

## 3. Consequences (結果)
- **メリット**:
  - **高いデータ信頼性**: AP試験のデータベース分野でも重要視される「正規化」や「トランザクション制御」の理論を忠実に実装できる。
  - **ポータビリティ**: 標準的な SQL および JPA を用いることで、DB製品（PostgreSQL, MySQL, Oracle 等）の切り替えが比較的容易になる。
  - **再現性**: マイグレーションツールにより、誰でも同じ状態のデータベース環境を即座に構築できる。
- **副作用 / 残存リスク**:
  - **N+1問題**: ORM の不適切な利用により、意図しない大量のクエリが発行され、パフォーマンスが低下するリスクがある。
  - **インピーダンス・ミスマッチ**: オブジェクト指向モデルとリレーショナルモデルの乖離を埋めるための設計コストが発生する。
