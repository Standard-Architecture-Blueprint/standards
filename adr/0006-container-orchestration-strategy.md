# ADR 0006: コンテナ・オーケストレーション戦略

- **Status**: Accepted
- **Date**: 2026-05-03
- **Deciders**: System Architect

## 1. Context (背景)
本プロジェクトでは、複数のリポジトリ（SoE, SoR, Platform 等）が並行して動作する。
開発者のローカル環境、検証環境、本番環境で「動く・動かない」の差異（Environment Discrepancy）を排除し、再現性の高い実行基盤が必要である。
また、将来的なクラウド移行（AWS, Azure 等）を容易にするため、特定のインフラに依存しない「ポータブルな実行単位」を定義する必要がある。

## 2. Decision (決定)
実行基盤として以下の技術および戦略を採用する。

- **Container Engine**: **Docker**
  - 理由: アプリケーションとその依存関係をパッケージ化するデファクトスタンダードであり、一貫した動作環境を提供できるため。
- **Orchestration (Development)**: **Docker Compose**
  - 理由: ローカル開発環境において、複数のマイクロサービス（Java, Angular, DB 等）をワンコマンドで立ち上げ、連携させるため。
- **Orchestration (Production Idea)**: **Kubernetes (K8s) Ready**
  - 理由: 将来的なスケーラビリティや自己修復（Self-healing）の要求に応えるため、K8s へのデプロイを想定したマニフェスト管理や、ステートレスな設計を標準とする。
- **Base Image**: **Alpine Linux / Distroless**
  - 理由: イメージサイズの軽量化と、不要なパッケージを排除することによる攻撃面の最小化（セキュリティ向上）。

## 3. Consequences (結果)
- **メリット**:
  - **環境の抽象化**: 「自分のPCでは動いた」という問題を根絶し、CI/CD パイプラインとの親和性が極めて高くなる。
  - **リソース効率**: 仮想マシン（VM）よりも軽量に実行でき、同一ホスト上で複数のサービスを効率よく稼働させられる。
  - **Infrastructure as Code (IaC)**: インフラ構成をコードで管理することで、構築の自動化と履歴管理が可能になる。
- **副作用 / 残存リスク**:
  - **学習コスト**: Dockerfile の最適化や、コンテナ間ネットワーク、ボリュームのマウント設定に関する知識が必要となる。
  - **管理の複雑性**: コンテナ数が増加するにつれ、一元的なログ管理や監視（Observability）の仕組みが不可欠となる。
