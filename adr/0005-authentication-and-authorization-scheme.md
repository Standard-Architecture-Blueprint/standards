# ADR 0005: 認証・認可スキームの策定（OAuth 2.0 / OIDC）

- **Status**: Accepted
- **Date**: 2026-05-03
- **Deciders**: System Architect

## 1. Context (背景)
SoE（フロントエンド）と SoR（バックエンド）が分離されたアーキテクチャでは、セッションベースの認証よりも、ステートレスなトークンベースの認証が適している。
また、将来的な外部システム連携やシングルサインオン（SSO）を見据え、独自実装によるセキュリティホールのリスクを排除し、業界標準のプロトコルを採用する必要がある。

## 2. Decision (決定)
認証・認可の仕組みとして、以下のデファクトスタンダードを採用する。

- **Protocol**: **OAuth 2.0 および OpenID Connect (OIDC)**
  - 理由: 認可（権限付与）と認証（本人確認）を分離して扱うための世界標準であるため。
- **Token Type**: **JWT (JSON Web Token)**
  - 理由: ステートレスにユーザー情報を伝搬でき、バックエンドでの検証を高速化するため。
- **Flow**: **Authorization Code Flow with PKCE**
  - 理由: SPA（Angular）において最も安全にトークンを取得できるフローであるため。
- **Implementation**: **Spring Security**
  - 理由: Java エコシステムにおいて最も堅牢かつ柔軟なセキュリティフレームワークであり、OIDC との親和性が高いため。

## 3. Consequences (結果)
- **メリット**:
  - **高い安全性**: 実証済みのプロトコルを用いることで、セッションハイジャックや CSRF などの脆弱性リスクを大幅に低減できる。
  - **拡張性**: 将来的に Google や GitHub 等の外部 ID プロバイダとの連携（ソーシャルログイン）が容易になる。
  - **標準知識の習得**: ここで扱う概念（Scope, Claim, Grant Type 等）は、あらゆるモダンなシステム開発に共通する基礎知識となる。
- **副作用 / 残存リスク**:
  - **実装の複雑さ**: プロトコルの理解が必要であり、初期の実装・設定コストが高い。
  - **トークンの管理**: JWT の有効期限（TTL）設計や、リフレッシュトークンの安全な保存方法について、慎重な検討が必要。
