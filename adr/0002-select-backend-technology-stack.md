# ADR 0002: バックエンド技術スタックの選定（Java / Spring Boot）

- **Status**: Accepted
- **Date**: 2026-05-03
- **Deciders**: System Architect

## 1. Context (背景)
SoR (Systems of Record) レイヤにおいて、データの整合性と長期的な保守性を担保するためには、型安全（Type-safe）であり、かつ広範なエコシステムを持つプログラミング言語およびフレームワークが必要である。
また、汎用的なエンジニアリングスキルとして、オブジェクト指向設計の原則（SOLID原則）やデザインパターンを深く学べる環境を選択する必要がある。

## 2. Decision (決定)
バックエンドの主要スタックとして以下を採用する。

- **Language**: **Java 17 (LTS)**
  - 理由: 強力な静的型付け、豊富なライブラリ、およびエンタープライズ領域における圧倒的なシェア。モダンな機能（Record, Switch式等）の活用による記述の簡略化。
- **Framework**: **Spring Boot 3.x**
  - 理由: Dependency Injection (DI) や AOP (アスペクト指向) といったアーキテクチャの根幹を学べる。また、Spring Security や Spring Data JPA 等のデファクトスタンダードなモジュール群との親和性。
- **Build Tool**: **Gradle**
  - 理由: Maven よりも柔軟な記述が可能であり、マルチモジュール構成の管理に適している。

## 3. Consequences (結果)
- **メリット**:
  - **堅牢性**: コンパイル時におけるエラー検知能力が高く、大規模開発でのバグ混入を抑制できる。
  - **学習の汎用性**: ここで学ぶ DI やデザインパターンは、他の言語（C#, TypeScript, Go 等）にも応用可能な基礎知識となる。
  - **市場価値**: 多くのエンタープライズシステムで採用されているスタックであり、技術的知見の再利用性が極めて高い。
- **副作用 / 残存リスク**:
  - 他の軽量言語（Node.js, Python 等）と比較して、起動速度（Cold Start）が遅く、メモリ消費量が多くなる傾向がある。
  - フレームワーク自体が巨大であるため、その全容を把握するための学習コストが初期段階で発生する。
