# 要件定義記述ルール (Requirement Definition Rules)

## 📌 概要
本ドキュメントは、プロジェクトにおける要件定義の標準的な記述方法を定める。AP試験の「共通フレーム（SLCP 2013）」に基づき、利害関係者要件を「機能要件」と「非機能要件」に分類し、後続の設計フェーズとのトレーサビリティを確保することを目的とする。

## 1. 機能要件 (Functional Requirements)
システムが「何をするか」を定義する。以下の要素を必須項目として記述すること。

- **業務フロー**: システム化対象業務の開始から終了までの流れ（AS-IS/TO-BE）。
- **機能一覧**: 画面、帳票、バッチ処理、外部システム連携などの具体的機能。
- **データ要件**: システムが扱う主要なエンティティ（実体）と、そのライフサイクル。
- **記述形式**:
  - 「～できること（SHALL）」という形式で、検証可能な最小単位で記述する。
  - 主語（誰が/何が）と目的語（何を）を明確にする。

## 2. 非機能要件 (Non-Functional Requirements)
システムが「どのような品質で動くか」を定義する。AP試験で頻出の **RASIS** および「非機能要件グレード」の6大項目に基づき記述する。

| 項目 | 定義内容の例 |
| :--- | :--- |
| **可用性 (Availability)** | 稼働時間（24/365）、目標復旧時間（RTO）、目標復旧時点（RPO）。 |
| **性能・拡張性 (Performance)** | 応答時間（レスポンスタイム）、スループット、最大同時接続数。 |
| **運用・保守性 (Maintainability)** | 計画停止の許容範囲、監視項目、バックアップ世代数。 |
| **移行性 (Portability)** | 現行システムからのデータ移行、切り戻し基準。 |
| **セキュリティ (Security)** | 認証・認可方式、暗号化（SSL/TLS）、監査ログ、脆弱性対策。 |
| **システム環境 (Environment)** | 準拠すべき法的規制、クラウド/オンプレミス等のインフラ制約。 |

## 3. 制約事項 (Constraints)
技術的、予算的、時間的な制限により変更不可能な条件。
- **技術制約**: 使用言語（Java 21）、フレームワーク（Spring Boot）の指定。
- **スケジュール制約**: 本番稼働開始日（カットオーバー）の遵守。

## 4. 品質基準 (Acceptance Criteria)
要件が「満たされた」と判断するための客観的基準。
- 全ての機能要件に対し、対応するテストケースが存在すること。
- 非機能要件で定義した数値目標（例：レスポンス3秒以内）をテスト環境で実証すること。

## 5. 追跡可能性 (Traceability) の維持
- 各要件には一意の **要件ID**（例：REQ-001）を付与する。
- 後続の設計書（FD/DD）およびテスト仕様書には、必ず対応する要件IDを明記し、要件漏れがないことを証明可能にする。
