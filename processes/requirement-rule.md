# 要件定義記述ルール (Requirement Definition Rules)

## 📌 概要
本ドキュメントは、プロジェクトにおける要件定義の標準的な記述方法を定める。AP試験の「共通フレーム（SLCP 2013）」に基づき、利害関係者要件から「システム要件」を導出し、機能・非機能の両面から後続工程とのトレーサビリティを確保することを目的とする。

## 1. 共通ルール：用語の定義 (Glossary)
リポジトリ間の論理的齟齬を排除するため、以下のルールを適用する。
- **用語集の参照**: 全ての要件記述は、プロジェクト共通の「用語辞書」に定義された名称を使用する。
- **一意性**: 同一の概念に異なる名称（シノニム）を与えない。異なる概念に同一の名称（ホモニム）を与えない。

## 2. 機能要件 (Functional Requirements)
システムが「何をするか」を定義する。
- **業務フロー**: 業務の開始から終了までの流れ（AS-IS/TO-BE）をMermaid図解で明文化する。
- **機能一覧**: 画面、帳票、バッチ、外部API連携等の具体的機能。
- **データ要件**: エンティティのライフサイクル（CRUD図）および主要属性。
- **記述形式**:
  - 「[アクター]は、[対象]を[操作]できること（SHALL）」という形式で記述する。
  - 単一の要件IDに対し、単一の検証可能な事象を記述する（分割不可能な最小単位）。

## 3. 非機能要件 (Non-Functional Requirements)
システムが「どのような品質で動くか」を定義する。AP試験の **RASIS** および「非機能要件グレード」に基づき記述する。

| 項目 | 定義内容の例 |
| :--- | :--- |
| **可用性 (Availability)** | 稼働時間、RTO/RPO、冗長化構成の要否。 |
| **性能・拡張性 (Performance)** | レスポンスタイム、スループット、将来のデータ増加予測。 |
| **運用・保守性 (Maintainability)** | 監視・バックアップ方針、障害通知基準。 |
| **移行性 (Portability)** | データ移行元、移行ツール、切り戻し要件。 |
| **セキュリティ (Security)** | 認証・認可(JWT/OAuth2)、暗号化、監査ログ、脆弱性診断。 |
| **システム環境 (Environment)** | 準拠法、インフラ制約（AWS/GCP/オンプレ等）。 |

## 4. 制約事項 (Constraints)
- **技術制約**: Java 21, Spring Boot 3, Angular 等の技術スタック指定。
- **外部制約**: 連携先システムのインターフェース仕様、法令遵守。

## 5. 品質基準 (Acceptance Criteria)
要件が「充足された」と判断するための客観的・定量的基準。
- **網羅性**: 要件IDとテストケースの1:Nの対応関係が証明されていること。
- **実証性**: 非機能要件の数値目標が負荷テスト等で実証されていること。

## 6. 追跡可能性 (Traceability) の維持
- 各要件に一意の **要件ID**（例：`REQ-001`）を付与する。
- 基本設計書(FD)以降、すべての成果物（コード、テスト、ADR）に要件IDを紐付け、変更時の影響範囲を特定可能にする。
