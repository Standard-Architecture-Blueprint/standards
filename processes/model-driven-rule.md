# モデリング・図解標準規約 (Model-Driven Rule)

## 📌 概要
本プロジェクトでは、仕様の可視化と共有の効率化のため、Mermaid.js を用いたモデリングを標準とする。AP試験の「構造化分析」および「オブジェクト指向設計」の理論に基づき、図解の抽象度と記法を統一することで、設計の「読み解き」の属人性を排除する。

## 1. 基本方針 (Fundamental Policy)
- **コードとしての図 (Diagram as Code)**: すべての図はリポジトリ内で Markdown 形式で管理し、Git による差分管理を可能にする。
- **シングル・ソース・オブ・トゥルース**: 設計図は常に実装の「最新の正解」を反映しなければならない。
- **抽象度の整合性**: 要件定義（概念）、基本設計（論理）、詳細設計（物理）の各フェーズで、適切な粒度の図を選択する。

## 2. 使用する図の種別と用途 (Diagram Types)
AP試験で問われる主要な UML および構造化手法に基づき、以下の 4 種を標準採用する。

| 図の名称 | 推奨フェーズ | 用途・記述内容 |
| :--- | :--- | :--- |
| **シーケンス図** | 基本/詳細設計 | オブジェクト間、またはサービス間の動的なメッセージのやり取り。 |
| **クラス図** | 詳細設計 | システムの静的な構造。ドメインモデル、依存関係、継承関係の定義。 |
| **ER図** | 基本設計 | データベースの物理構造。リレーション、主キー、外部キーの定義。 |
| **状態遷移図** | 基本/詳細設計 | 複雑なライフサイクル（例：注文ステータス）を持つオブジェクトの状態変化。 |

## 3. Mermaid 記法標準 (Style Guide)

### 3.1. 命名規則
- **PascalCase**: クラス名、エンティティ名、アクター名。
- **camelCase**: メソッド名、属性名、変数名。
- **SNAKE_CASE**: 定数、状態（ステータス）定義。

### 3.2. 記述の制約
- **一方向性**: 図の情報の流れは、原則として「上から下」または「左から右」に統一する。
- **依存の可視化**: クラス図では `+--`（継承）や `-->`（依存）を適切に使い分け、AP試験の「凝集度」と「結合度」の概念に基づいた低結合な設計を証明する。
- **コメント**: 複雑な分岐ロジックがある場合は、`Note` 機能を用いて補足説明を加える。

## 4. 各工程におけるモデリングの粒度

### 4.1. 上流工程（概念モデル）
- 詳細な属性（型情報など）は省略し、アクターとシステム、またはドメイン間の主要な関係性（境界付けられたコンテキスト）のみを記述する。

### 4.2. 下流工程（物理モデル）
- 詳細設計では、型定義（`String`, `Int` 等）や可視性（`+`, `-`）、多重度（`1..*`）を明記し、そのまま実装コードへと変換（フォワードエンジニアリング）できるレベルまで記述する。

## 5. メンテナンスルール
- **レビュー対象**: プルリクエスト（PR）時、コードの変更に付随して対応する Mermaid 図が更新されているかを確認する。
- **廃止**: 実装と乖離し、更新コストが見合わなくなった図は、コメントアウトするのではなく速やかに削除し、常に「真実」のみを掲載する。
