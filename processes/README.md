# 1. Management & Processes (管理プロセス規約)

## 📌 概要
本ディレクトリでは、Standard Architecture Blueprint (SAB) プロジェクトにおける開発ライフサイクル全体の「進め方」を定義する。AP試験の「共通フレーム（SLCP 2013）」に準拠し、企画から要件定義、開発、運用に至る各プロセスの透明性と一貫性を確保することを目的とする。

## 🏛️ 管理資産（ドキュメント構成）

### [requirement-rule.md](./requirement-rule.md)
**要件定義の記述ルール**
- **機能要件**: 利用者が実現したい業務機能の定義方法。
- **非機能要件**: AP試験の品質特性（RASIS）および「非機能要件グレード」に基づいた、可用性、性能、拡張性、セキュリティの定義基準。
- **目的**: 開発中盤での要件漏れ（スコープクリープ）を防止し、検証可能な要件を定義する。

### [model-driven-rule.md](./model-driven-rule.md)
**モデリングおよび図解の標準記法**
- **Mermaidの活用**: クラス図、シーケンス図、ER図、状態遷移図の記法標準。
- **抽象化レベル**: 概念モデル（上流）から実装モデル（下流）への落とし込み方。
- **目的**: 設計書の記述言語を統一し、レビューの効率と設計の追跡可能性（Traceability）を高める。

### [release-process.md](./release-process.md)
**承認フローと変更管理**
- **ワークフロー**: 開発環境から本番環境へのプロモーション（昇格）手順。
- **変更管理**: ITILおよびAP試験の「サービスマネジメント」に基づく変更要求（CR）の承認プロセス。
- **目的**: 未承認の変更によるシステムの不安定化を防ぎ、構成の完全性を維持する。

## 📝 開発ライフサイクル (SDLC) の原則

1. **合意形成の重視**: 各工程の成果物は、定義された承認者によるレビュー（インスペクション/ウォークスルー）を経て、次の工程へ進むこと。
2. **構成管理の徹底**: 全ての成果物（ドキュメント含む）はGitでバージョン管理し、ソースコードとの整合性を常に維持すること。
3. **継続的改善**: プロセスに不備が見つかった場合、速やかに本ディレクトリの規約を更新し、組織知として蓄積すること。
