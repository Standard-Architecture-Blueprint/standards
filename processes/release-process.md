# リリース・変更管理規約 (Release & Change Management Process)

## 📌 概要
本ドキュメントは、プロジェクトにおける変更の承認および本番環境へのリリースプロセスを定める。AP試験の「サービスマネジメント」における「変更管理」と「リリース管理」の原則に基づき、システムの安定稼働と構成の完全性を担保することを目的とする。

## 1. 変更管理 (Change Management)
意図しない変更によるサービス停止を防ぐため、すべての変更は以下のフローを遵守しなければならない。

- **変更要求 (Request for Change: RFC)**: 課題（Issue）またはプルリクエスト（PR）をもって変更内容、理由、影響範囲を明文化する。
- **承認 (Approval)**: リポジトリ管理者（システムアーキテクト）によるレビューを受け、承認を得る。
- **追跡可能性 (Traceability)**: すべての変更は「要件ID」または「Issue ID」と紐付けられ、監査時に「なぜこの変更が行われたか」を説明可能にする。

## 2. リリースサイクルと環境定義
AP試験の「システム開発ライフサイクル」および継続的デリバリ（CD）の考え方に基づき、環境を分離する。

| 環境名称 | 用途 | リリース基準 |
| :--- | :--- | :--- |
| **Development** | 開発者による機能実装・単体テスト | コードのプッシュにより自動反映。 |
| **Staging** | 結合テスト・受入テスト (UAT) | `main` ブランチへのマージ、および自動テスト全パス。 |
| **Production** | 本番運用環境 | ステージングでの承認完了、および管理者によるリリース署名。 |

## 3. リリースフロー (Release Workflow)

### 3.1. プルリクエスト (PR) 基準
以下の「クオリティゲート」を通過しない限り、上位環境へのリリースは許可されない。
- **静的解析**: Lint および静的解析ツールで「致命的エラー」がゼロであること。
- **テスト網羅率**: `testing/test-strategy.md` で定義された C0/C1 網羅率を満たしていること。
- **ピアレビュー**: 自分以外の開発者（または管理者）による承認（Approve）を得ていること。

### 3.2. デプロイメント手順
- **CI/CD パイプライン**: 原則として手動デプロイを禁止し、GitHub Actions 等の自動化ツールを用いる。
- **イミュータブル・インフラストラクチャ**: コンテナ（Docker等）を活用し、環境ごとの差異による不具合を排除する。

## 4. 切り戻し（ロールバック）方針
リリース後に重大な障害が発生した場合の「非常口」を事前に定義する。
- **判断基準**: 本番反映後、重要機能に致命的な影響がある場合、または回復に 30 分以上を要すると判断された場合。
- **手順**: 前バージョンのコンテナイメージまたは Git タグへの即時切り戻しを実施する。
- **事後分析**: 切り戻し発生時は、AP試験の「問題管理」に基づき根本原因分析（RCA）を行い、再発防止策を講じる。

## 5. 証跡管理 (Audit Trail)
システム監査に対応するため、以下の記録を永続的に保持する。
- **リリースログ**: 誰が、いつ、どのバージョンのコードをデプロイしたかの記録。
- **承認記録**: プルリクエスト内のコメントおよび Approve 履歴。
- **検証記録**: リリース直前のテスト実行結果エビデンス。
