# Backup & Recovery Strategy (バックアップ・リカバリ戦略)

## 1. 📌 概要
本ドキュメントは、システム障害、操作ミス、または災害時において、データを保護し、サービスを目標時間内に復旧させるための指針を定義する。AP試験で扱われる「バックアップ/リカバリ手法」に基づき、データの整合性と機密性を維持する。

## 2. 🎯 目標指標 (BCP指標)
複数のリポジトリ全体の信頼性を支えるため、以下の指標を基準とする。

- **RPO (Recovery Point Objective: 目標復旧時点)**: 
  - 障害発生時から最大で何時間前のデータ状態まで戻すか。
  - **基準**: 24時間以内（重要データはポイントインタイムリカバリにより数分以内）。
- **RTO (Recovery Time Objective: 目標復旧時間)**: 
  - 障害発生からサービスを再開するまでの目標時間。
  - **基準**: 4時間以内（インフラの自動構築とバックアップからの自動リストアを活用）。

## 3. 📂 バックアップ対象と手法
AP試験の「ストレージ管理」に基づき、対象ごとに最適な手法を選択する。

| 対象 | 手法 | 頻度 | 保存先 |
|:---|:---|:---|:---|
| **データベース (SoR)** | スナップショット + ログ転送 | 毎日フル + 逐次差分 | 別リージョンのオブジェクトストレージ |
| **ファイルストレージ** | バージョニング + レプリケーション | リアルタイム | オブジェクトストレージ (S3等) |
| **システム設定 (IaC)** | Git リポジトリ管理 | 随時 | GitHub/GitLab ( engineering/ 参照) |
| **コンテナイメージ** | イメージレジストリ保存 | ビルド毎 | コンテナレジストリ |

## 4. 🔄 リカバリ手順 (Restore Process)
有事の際に迷わず実行できるよう、手順を標準化する。

1. **損害評価**: データの欠損状況と、どの時点（PIT）まで遡るべきかを特定する。
2. **環境確保**: `engineering/environment-setup.md` に基づき、クリーンなインフラ環境をプロビジョニングする。
3. **データ流し込み**: バックアップアーカイブからデータをリストアする。
4. **整合性チェック**: アプリケーションレイヤでデータの論理的な整合性（外部キー制約等）を検証する。
5. **サービス再開**: 段階的にトラフィックを流し、正常性を確認する。

## 5. 🛠️ 定期的な復旧訓練
- **訓練の実施**: 年に最低1回、バックアップデータから実際に環境を復旧できるかテストを行う。
- **手順の更新**: 訓練結果に基づき、本ドキュメントおよび自動化スクリプトを改善する。

## 6. 🔐 バックアップの保護
- **暗号化**: 保存されるすべてのバックアップデータは、AES-256等の強力な暗号化を施す。
- **不変性 (Immutability)**: 内部不正やランサムウェア対策として、一定期間削除不能なオブジェクトロック（WORM特性）を設定する。
- **物理的隔離**: 地理的に離れたリージョン（遠隔地）へバックアップを保管し、広域災害に備える。
