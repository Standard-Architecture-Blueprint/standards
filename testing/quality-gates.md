# 品質ゲート基準 (Quality Gates)

## 📌 概要
本ドキュメントは、Standard Architecture Blueprint (SAB) におけるコード品質の最終判定基準（クオリティ・ゲート）を定める。AP試験の「品質管理」および「ソフトウェア構成管理」に基づき、人的ミスを排し、常に安定したシステム状態を維持するためのマージ条件を定義することを目的とする。

## 1. 構成管理と統合 (Configuration Management)
AP試験の「構成管理」に基づき、メインブランチの整合性を保護するためのゲートを設ける。

- **プルリクエスト（PR）の必須化**: 
  - `main` または `develop` ブランチへの直接コミットを禁止する。
  - 全ての変更は PR を経由し、後述する自動・手動の検証を通過しなければならない。
- **追跡可能性（Traceability）**: 
  - 各 PR には対応するチケット（または要件ID）の紐付けを必須とし、「何のための変更か」を明確にする。

## 2. 自動クオリティ・ゲート (Automated Gates)
CI（継続的インテグレーション）パイプラインで自動判定される基準。

- **ビルド成功**: 
  - 全てのコードが正常にコンパイル/トランスパイルされ、依存関係に不整合がないこと。
- **テスト合格率 100%**: 
  - 定義された全ての単体テスト（UT）および結合テスト（IT）がエラーなしで完了すること。
- **命令網羅率 (C0 Coverage)**: 
  - 80% 以上のカバー率を維持していること。基準を下回る場合はマージをブロックする。
- **静的解析 (Linter / Static Analysis)**: 
  - 命名規則、コードの複雑度（循環的複雑度）が許容値以内であること。
  - 重要度「High」以上の警告が 0 件であること。

## 3. レビュー・ゲート (Human Review)
AP試験の「レビュー技法」に基づき、専門家（アーキテクト・リード）による定性的な検証を行う。

- **ピアレビューの実施**: 
  - 最低1名の「Approve」をマージの必須条件とする。
- **レビュー観点**:
  - **設計の整合性**: `design/` で定義されたアーキテクチャ・パターンに準拠しているか。
  - **保守性**: マジックナンバーの排除、重複コードの抽出が行われているか。
  - **セキュリティ**: `engineering/security-standard.md` に則り、脆弱性のある記述が含まれていないか。

## 4. セキュリティ・ゲート (Security Gates)
AP試験の「セキュリティ管理」に基づく強制基準。

- **脆弱性スキャン (SCA)**: 
  - ライブラリ等の依存関係に、既知の脆弱性（CVE）が含まれていないこと。
- **シークレット・スキャン**: 
  - ソースコード内に API キーやパスワード等の機密情報が混入していないこと。

## 5. 例外的なゲート緩和 (Emergency Procedures)
緊急障害対応時などでゲートを迂回する場合は、以下のプロセスを必須とする。
- 承認者による事後承認の記録。
- 暫定対応後の「恒久対策チケット」の発行と、速やかな技術的負債の解消。
