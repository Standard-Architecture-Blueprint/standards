# 5. Quality Assurance & Testing (品質保証とテスト)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) プロジェクトにおけるテスト戦略と品質保証基準を定める。AP試験の「ソフトウェアテスト」の体系に基づき、13のリポジトリにおいて「客観的に証明可能な品質」を維持し、リリース後の信頼性を担保することを目的とする。

## 🏛️ 管理資産（品質保証ガイドライン）

### [test-strategy.md](./test-strategy.md)
**C0/C1網羅率・テスト範囲**
- **網羅率基準**: 命令網羅（C0）および判定条件網羅（C1）の最小合格ラインの設定。
- **テストピラミッド**: 単体テスト（UT）を厚くし、結合（IT）・E2Eの比率を最適化する戦略。
- **目的**: テストの「漏れ」を論理的に排除し、リファクタリング耐性の高いコードベースを維持する。

### [quality-gates.md](./quality-gates.md)
**PR承認基準・静的解析ルール**
- **マージ基準**: CI（継続的インテグレーション）での自動テスト通過、および静的解析（Linter/Security Scan）のパス。
- **レビュー方針**: コードの可読性、設計原則（DIP/SRP等）への準拠を確認するレビュープロセス。
- **目的**: 人的ミスをシステムで防ぎ、13のリポジトリで「均一なコード品質」を強制する。

### [evidence-standard.md](./evidence-standard.md)
**証跡（トレーサビリティ）標準**
- **テスト結果の記録**: 自動テストレポート、手動テストのエビデンス保存形式。
- **トレーサビリティ**: 要件（RD）とテストケースを紐付け、全ての要件が検証済みであることを証明する。
- **目的**: AP試験の「システム監査」に耐えうる証跡を管理し、変更の影響範囲特定を迅速化する。

## 🧪 テストの基本原則 (Testing Principles)
AP試験の「ソフトウェアテスト手法」に基づき、以下の原則を遵守せよ。

- **不具合の偏在 (Defect Clustering)**: 特定の複雑なモジュールにバグが集中することを想定し、重要機能には重点的なテスト（境界値分析等）を行う。
- **全数テストの不可能**: テストは「欠陥があること」を証明できるが「欠陥がないこと」は証明できない。よって、リスクベースで優先順位を決定する。
- **早期テスト (Early Testing)**: 要件定義・設計段階からテスト観点を盛り込み、後工程での手戻りコストを最小化する。

## 📝 品質レビューの合否基準
テスト成果物およびプルリクエストは、以下のチェックリストを満たす必要がある。

- [ ] 自動テストの命令網羅率（C0）が 80% 以上を記録しているか。
- [ ] 境界値分析に基づいた異常系テストケースが含まれているか。
- [ ] 静的解析ツールによって「High/Critical」な警告が 0 件となっているか。
- [ ] テスト結果が要件ID（RD）に関連付けられ、追跡可能（Traceable）か。
