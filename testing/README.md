# 5. Testing Standards (テスト標準・品質管理)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) におけるテスト戦略と品質保証基準を定める。AP試験の「ソフトウェアテスト」の体系に基づき、開発工程とテスト工程を密接に連携させ、欠陥の早期発見と再デプロイ時の回帰テスト（レジリエンシー）を担保することを目的とする。

## 🏛️ テスト工程の定義 (V字モデルの適用)
AP試験の「開発プロセス」に基づき、以下のテストレベルを定義し、各設計工程と対応付ける。

| テストレベル | 対応する設計工程 | 実施内容の要点 |
| :--- | :--- | :--- |
| **単体テスト (UT)** | 詳細設計 / 実装 | モジュール（クラス/関数）単位の論理検証。ホワイトボックス法。 |
| **結合テスト (IT)** | 内部設計 | インターフェース間のデータ受け渡し、サブシステム間の連携検証。 |
| **システムテスト (ST)** | 外部設計 / 要件定義 | 非機能要件（性能、負荷、セキュリティ）を含むシステム全体の検証。 |
| **受入テスト (UAT)** | 要件定義 / ビジネス要件 | ユーザー視点での業務フロー完遂および運用要件の確認。 |

## 🧪 テスト手法と網羅性
AP試験で問われるテスト理論に基づき、定量的かつ論理的なテストを実施する。

### 1. ホワイトボックステスト (UT中心)
- **制御の流れの網羅**: 以下の網羅基準をコードの重要度に応じて適用する。
    - **命令網羅 (C0)**: 全ての命令を少なくとも1回は実行する（標準: 80%以上）。
    - **判定条件網羅 (C1)**: 全ての分岐（True/False）を少なくとも1回は実行する。
- **静的解析**: 実行前に Linter や型チェックにより、構文エラーや潜在的なバグ（メモリリーク、未定義変数）を排除する。

### 2. ブラックボックステスト (IT/ST中心)
- **同値分割**: 有効な入力値と無効な入力値のグループから代表値を選択する。
- **境界値分析**: 条件の境界（最大値、最小値、その前後）を重点的に検証し、オフバイワンエラーを検出する。

## 🔄 テスト自動化と CI/CD
AP試験の「ソフトウェアメンテナンス」を効率化するため、自動化を原則とする。

- **回帰テスト (Regression Test)**: コード変更時に既存機能が損なわれていないかを自動実行で確認する。
- **モックとスタブの活用**: 外部APIやデータベースに依存せず、`engineering/design-patterns.md` で定義した DI を活用してテストの独立性を高める。
- **継続的テスト**: プルリクエスト作成時にパイプライン上でテストを自動実行し、品質のゲートキーパーとする。

## ✅ 品質評価指標 (Quality Metrics)
AP試験の「品質管理」に基づき、以下のメトリクスを計測・評価する。

- **テスト密度**: プログラム規模に対するテスト項目数が適切か（過少・過多の判定）。
- **欠陥密度**: プログラム規模に対するバグ発見数が予測範囲内か。
- **バグ収束曲線**: テストの進捗とバグ発見数の推移を監視し、品質が安定したことを論理的に判断する。
