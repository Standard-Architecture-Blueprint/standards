# 証跡標準 (Evidence Standard)

## 📌 概要
本ドキュメントは、Standard Architecture Blueprint (SAB) におけるテスト証跡（エビデンス）の記録基準を定める。AP試験の「システム監査」および「品質管理」に基づき、テスト実施の事実と結果の妥当性を客観的に証明し、不具合発生時の原因究明を迅速化することを目的とする。

## 1. 証跡の基本原則
AP試験の「システム監査基準」に準じ、証跡は以下の特性を備えなければならない。

- **不変性**: 記録された証跡は、後から改ざんできない状態で保存されること（CI/CDログの保存など）。
- **再現性**: 証跡に基づき、第三者が同じ環境で同一のテスト結果を再現できる情報が含まれていること。
- **追跡可能性 (Traceability)**: 要件定義（RD）→ 設計書 → テストケース → テスト結果 が一貫して紐付いていること。

## 2. 自動テストの証跡 (Automated Test Evidence)
CIパイプラインで実行されるテストについては、ツールが生成するレポートを公式な証跡とする。

- **実行ログ**: 実行日時、実行者（CI Bot）、成功/失敗のステータス、詳細なスタックトレース。
- **カバレッジレポート**: `test-strategy.md` で定めた網羅率（C0/C1）を満たしていることを示す統計データ。
- **保存期間**: 少なくとも対象リリースの運用が終了するまで、リポジトリの履歴または外部ストレージに保管する。

## 3. 手動テストの証跡 (Manual Test Evidence)
UI/UXテストや受入テストなど、手動で実施が必要な項目に関する標準。

- **期待値と実績値**: 各テストステップに対し、「期待される動作」と「実際の動作」を対比して記録する。
- **視覚的エビデンス**: 
  - 画面キャプチャ（重要な遷移、データの登録完了画面、エラーメッセージ等）。
  - キャプチャには、実施日時が特定できる情報（システム時計等）を含めることが望ましい。
- **実行環境情報**: ブラウザの種類・バージョン、OS、テストデータのID、使用した端末情報。

## 4. トレーサビリティ・マトリクスの管理
AP試験の「プロジェクトマネジメント」における品質管理手法に基づき、以下のマトリクスを維持する。

- **要件紐付け**: 全てのテストケースは、対応する「要件ID」を明記しなければならない。
- **網羅性の証明**: 未実施のテストケースや、要件に対応していないテストケースが存在しないことを定期的に確認する。

## 5. 異常系の証跡管理
- **不具合票との連携**: テストで検出されたバグについては、証跡（ログ、スクリーンショット）を不具合管理票（GitHub Issues等）に直接添付し、修正後の再テスト結果と対比させる。
- **デバッグ情報の分離**: 開発時のログと証跡としてのログを区別し、機密情報が含まれないよう注意を払う。
