# 4. Engineering Standards (実装・エンジニアリング標準)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) プロジェクトにおける実装・テスト・運用の標準化指針を定める。AP試験の「ソフトウェア開発」および「品質管理」に基づき、13のリポジトリにおける「コードの品質」と「開発の継続性」を担保することを目的とする。

## 🏛️ 管理資産（エンジニアリングガイドライン）

### [coding-standard.md](./coding-standard.md)
**命名・BaseException・ログの型**
- **コーディング規約**: 言語ごとのデファクトスタンダード（Google/Oracle等）に基づく命名・書式の統一。
- **例外処理の標準化**: `BaseException` を基底とした例外体系と、エラーコードの定義。
- **目的**: ソフトウェアの「保守性」を高め、属人性を排除した一貫性のあるコードを生成する。

### [design-patterns.md](./design-patterns.md)
**DI・共通実装パターン**
- **DI (Dependency Injection)**: コンストラクタ注入による疎結合なコンポーネント構成。
- **定石パターンの適用**: Repository, Strategy, Factory 等、AP試験頻出パターンの実装標準。
- **目的**: モジュールの「独立性」を高め、変化に強い内部構造を維持する。

### [security-standard.md](./security-standard.md)
**JWT・秘密情報管理**
- **認証・認可**: JWT のクレーム設計および検証プロセスの標準化。
- **機密情報保護**: 環境変数や Vault を利用した秘密情報の分離（ハードコードの厳禁）。
- **目的**: 「情報セキュリティ」基準を実装レベルで強制し、システム全体の堅牢性を確保する。

## 🛠️ エンジニアリング原則 (Engineering Principles)
AP試験の「ソフトウェアライフサイクル」の評価指標に基づき、以下の原則を遵守せよ。

- **フェイルファスト (Fail Fast)**: コンパイル時、または CI 時の自動テストで早期にエラーを検出し、下流工程への不具合流出を防止する。
- **コードの自己記述性**: 過度なコメントに頼らず、命名と構造だけでロジックが理解できる「Clean Code」を目指す。
- **継続的なリファクタリング**: ソフトウェア保守の概念に基づき、技術的負債を蓄積させず、常にコードを最適な状態に保つ。
- **セキュリティ・バイ・デザイン**: コーディング段階から脆弱性（インジェクション対策、サニタイジング等）を排除する。

## ✅ 定量的品質目標 (Quality Metrics)
AP試験の「品質管理」におけるメトリクスに基づき、以下の目標値を設定する。

- **テスト網羅率 (Coverage)**: 命令網羅（C0）80%以上、判定条件網羅（C1）を重要ロジックで適用。
- **静的解析警告**: 重要（High/Critical）な警告は「0件」であることをマージ条件とする。
- **レビュー承認数**: 全てのプルリクエストは、最低1名の承認（Approve）を必須とする。
