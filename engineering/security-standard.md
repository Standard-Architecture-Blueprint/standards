# セキュリティ標準 (Security Standard)

## 📌 概要
本ドキュメントは、Standard Architecture Blueprint (SAB) における情報セキュリティの実装基準を定める。AP試験の「セキュアプログラミング」および「ネットワークセキュリティ」に基づき、脆弱性を設計・実装段階から排除（セキュリティ・バイ・デザイン）することを目的とする。

## 1. 認証・認可 (Authentication & Authorization)
AP試験の「利用者認証」および「アクセス制御」の知識体系に基づく標準。

- **JWT (JSON Web Token) の運用**:
    - **署名アルゴリズム**: `RS256`（非対称鍵）などの強固なアルゴリズムを使用し、改ざんを防止する。
    - **ステートレス認証**: サーバー側にセッションを持たず、トークン内で完結する認証を行う。
    - **有効期限 (exp)**: アクセストークンは短命（例: 15分〜1時間）に設定し、盗難リスクを低減する。
- **RBAC (Role-Based Access Control)**:
    - ユーザーの役割（Admin, Editor, Viewer等）に基づき、APIエンドポイントへのアクセス権限を厳格に制御する。
    - 「最小権限の原則」に従い、業務上必要な権限のみを付与する。

## 2. 秘密情報の管理 (Secrets Management)
AP試験の「資産管理」および「情報漏洩対策」に基づく機密情報の取り扱い。

- **ソースコードへのハードコード禁止**:
    - APIキー、DB接続パスワード、暗号化鍵などをコード内に直接記述することを厳禁とする。
- **環境変数の利用**:
    - 秘密情報は実行環境（OSの環境変数）または専用の秘密情報管理サービス（AWS Secrets Manager等）から注入する。
- **.gitignore の徹底**:
    - `.env` ファイルや認証鍵ファイルが誤ってリポジトリにコミットされないよう、厳重に管理する。

## 3. 防御的プログラミング (Defensive Programming)
AP試験で問われる「Webアプリケーションの脆弱性攻撃」に対する直接的な防御策。

- **入力値バリデーション (Sanitizing)**:
    - 全ての外部入力に対し、型、長さ、形式をホワイトリスト形式で検証する。
    - SQLインジェクション対策として、必ず「プレースホルダ（静的プリペアードステートメント）」を使用する。
- **クロスサイト・スクリプティング (XSS) 対策**:
    - 出力時に適切なエスケープ処理を行う。また、Content Security Policy (CSP) ヘッダーを活用する。
- **安全な通信 (TLS)**:
    - 全ての通信は TLS 1.3 以上で暗号化し、HSTS（HTTP Strict Transport Security）を有効化して常時HTTPS化を強制する。

## 4. 暗号化とハッシュ化 (Cryptography)
AP試験の「暗号技術」に基づくデータの保護。

- **パスワードの保存**:
    - 平文での保存は厳禁。`Argon2` や `BCrypt` などのソルト付きハッシュアルゴリズムを使用し、ストレッチングを行う。
- **機密データの暗号化**:
    - DB内の機密データ（個人情報等）は、AES-256 等の標準的な暗号化アルゴリズムを用いて保護する。

## 5. セキュリティ・レビュー (Security Review)
- **依存関係の脆弱性診断**: CI/CDパイプラインにおいて、ライブラリの脆弱性を自動スキャン（SCAツール等）し、既知の脆弱性（CVE）を含むコードのデプロイを阻止する。
- **エラーメッセージの制御**: 攻撃者にヒントを与えないよう、詳細なスタックトレースやDBエラーを本番環境のレスポンスに含めない。
