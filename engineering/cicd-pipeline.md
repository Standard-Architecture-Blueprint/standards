# CI/CD Pipeline Standards (自動化パイプライン規約)

## 1. 📌 概要
本ドキュメントは、継続的インテグレーション（CI）および継続的デリバリー（CD）の自動化フローを定義する。AP試験における「変更管理」と「リリース管理」の知識に基づき、迅速かつ安全なリリースサイクルを実現する。

## 2. 🏗️ CI (Continuous Integration) フロー
各リポジトリへの Push および Pull Request (PR) 作成をトリガーとして実行する。

1. **Static Analysis (静的解析)**:
   - Lintチェック、型チェックを行い、コーディング規約（`design/`）への準拠を確認する。
2. **Unit / Integration Testing**:
   - `testing-strategy.md` に基づく自動テストを実行する。
3. **Vulnerability Scan (脆弱性診断)**:
   - 依存ライブラリに既知の脆弱性（CVE）が含まれていないかスキャンする。
4. **Build & Artifact Generation**:
   - コンテナイメージ（Docker Image）をビルドし、イメージレジストリへ格納する。

## 3. 🚀 CD (Continuous Delivery) フロー
CIが成功し、承認（Approve）されたコードを各環境へ反映する。

- **Staging (検証環境)**:
  - `main` ブランチへのマージをトリガーに自動デプロイする。
  - SoIレイヤとの連携や、エンドツーエンドの業務シナリオテストを実施する場とする。
- **Production (本番環境)**:
  - 承認者による手動トリガー（Approval Gate）を経てデプロイする。
  - ブルーグリーンデプロイメントまたはカナリアリリースを検討し、ダウンタイムを最小化する。

## 4. 🛠️ パイプライン構成の要件
### 4.1. 冪等性 (Idempotency) の確保
- 同じバージョンの成果物（Artifact）であれば、何度デプロイしても同じ状態になるように IaC（Infrastructure as Code）を用いる。

### 4.2. 環境変数の管理
- 接続先DBやAPIキーなどの機密情報は、コードに含めず、CI/CDツールの「Secrets」管理機能を使用してデプロイ時に注入する。

### 4.3. アーティファクトの不変性 (Immutability)
- 一度ビルドされたコンテナイメージは、環境（検証・本番）を跨いでも再ビルドせず、同じイメージタグを昇格（Promote）させて使用する。

## 5. 🚨 パイプライン失敗時の対応
- CI/CDが失敗した場合、そのブランチのマージは原則禁止する。
- 失敗通知（Slack/Teams等）を即座に開発者へ飛ばし、修正を最優先タスクとする。

## 6. 📊 メトリクスの監視
AP試験の「開発プロセス改善」に基づき、以下の 4 Keys メトリクスを意識する。
- **デプロイ頻度**: どのくらいの頻度でリリースできているか。
- **変更のリードタイム**: コードを書いてからリリースされるまでの時間。
- **変更失敗率**: デプロイの結果、障害が発生した割合。
- **復元時間 (MTTR)**: 障害発生からサービスが復旧するまでの時間。
