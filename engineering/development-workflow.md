# Development Workflow (開発ワークフロー規約)

## 1. 📌 概要
本ドキュメントは、チーム全体で一貫した開発リズムを維持し、コードの品質を担保するためのプロセスを定義する。構成管理の徹底により、複数のリポジトリ間での不整合を防ぐ。

## 2. 🌿 ブランチ戦略 (GitHub Flow)
シンプルさとリリース速度を重視し、**GitHub Flow** を採用する。

- **`main` ブランチ**:
  - 常にデプロイ可能で、最新かつ安定したコードが保持される。
  - 直接のコミットは厳禁とする。
- **`feature/` ブランチ**:
  - 新機能開発、バグ修正、タスクごとに `main` から分岐する。
  - 命名規則: `feature/{issue-number}-{summary}`
  - 作業完了後、`main` へ Pull Request (PR) を作成する。

## 3. 🚀 開発のサイクル
1. **Issue 作成**: 作業内容をタスク単位（Issue）で定義する。
2. **ブランチ作成**: `main` から最新のコードを取得し、作業用ブランチを作成する。
3. **コーディングとテスト**: ローカルで実装し、単体テストをパスさせる。
4. **Pull Request (PR) 作成**: 作業内容、テスト結果、影響範囲を記載して PR を出す。
5. **レビューと修正**: ピアレビューを受け、指摘事項を反映する。
6. **マージ**: CIがパスし、Approve を得たら `main` へマージする。

## 4. 📝 コミットメッセージ規約
AP試験の「ソフトウェア保守」の観点から、変更履歴を追いやすくするため **Conventional Commits** に準拠する。

**形式**: `<type>(<scope>): <subject>`

- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの動作に影響しない修正（フォーマット等）
- `refactor`: リファクタリング（機能追加やバグ修正を含まない）
- `test`: テストコードの追加・修正

**例**: `feat(auth): add OIDC login flow`

## 5. 👀 コードレビュー基準
レビューは単なる間違い探しではなく、知識共有と品質維持の場とする。

- **ロジックの正当性**: 設計指針（`design/`）に準拠しているか。
- **テスタビリティ**: 単体テストが適切に書かれているか。
- **可読性**: 命名は適切か、複雑すぎるロジックはないか。
- **セキュリティ**: SQLインジェクションや機密情報のハードコードがないか。

## 6. 🛠️ 開発ツール標準
- **IDE**: IntelliJ IDEA または VS Code (推奨拡張機能を共有)
- **Linter / Formatter**: 
  - Java: Google Java Style Guide
  - TypeScript: ESLint / Prettier
- **Git Client**: コマンドライン または GUIツール（Fork/GitKraken等）
