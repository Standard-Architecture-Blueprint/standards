# デザインパターン適用指針 (Design Patterns)

## 📌 概要
本ドキュメントは、Standard Architecture Blueprint (SAB) における共通実装パターンの適用基準を定める。AP試験の「ソフトウェア構造設計」における評価指標（結合度の低減、強度の向上）に基づき、メンテナンス性の高いコード構造の維持を目的とする。

## 1. 依存性注入 (Dependency Injection: DI)
AP試験の「モジュールの独立性」および「インターフェースへの依存」を具現化する最重要パターン。

- **Constructor Injection**: 
    - クラスが必要な依存オブジェクト（ServiceやRepository）は、すべてコンストラクタ経由で受け取る。
    - インスタンス生成時に依存関係が確定するため、不変性（Immutable）とテスト容易性が担保される。
- **Interface-based Programming**: 
    - 具体的なクラスではなく、インターフェースに対してプログラミングを行う。
    - これにより、単体テスト時にモック（Mock）やスタブへの差し替えが可能になり、テストの独立性が向上する。

## 2. GoF デザインパターンの標準採用
AP試験の「オブジェクト指向設計」における定石パターンのうち、マイクロサービス開発で頻出するものを標準化する。

### 2.1. 生成に関するパターン (Creational Patterns)
- **Factory パターン**:
    - オブジェクトの生成ロジックをカプセル化し、呼び出し側が具象クラスの詳細を知らなくて済むようにする。
- **Builder パターン**:
    - 設定項目が多いドメインモデルや DTO を構築する際、可読性と不変性を両立させる。

### 2.2. 構造に関するパターン (Structural Patterns)
- **Adapter パターン**:
    - 外部ライブラリやレガシーなインターフェースを、システム標準の形式（Interface）に変換して適合させる。
- **Facade パターン**:
    - 複雑なサブシステムの呼び出しを簡素なインターフェースで包み、利用側の複雑度を下げる。

### 2.3. 振る舞いに関するパターン (Behavioral Patterns)
- **Strategy パターン**:
    - 業務ルール（例：割引計算、配送方法の選択）をアルゴリズムとして分離し、実行時に動的に切り替え可能にする。
- **Template Method パターン**:
    - 基本的な処理フローをスーパークラスで定義し、具体的なステップをサブクラスで実装することで、ロジックの重複を排除する。

## 3. ドメイン駆動設計 (DDD) 由来のパターン
AP試験の「内部設計」をより現代的に発展させた、データアクセスの標準パターン。

- **Repository パターン**:
    - データストレージ（DBやAPI）へのアクセスを抽象化し、ドメイン層が「データの保存先がどこか」という詳細に依存しないようにする。
- **Value Object (値オブジェクト)**:
    - 単なる文字列や数値ではなく、型（Class）として定義することで、バリデーションとドメインロジックを型の中に封じ込める。

## 4. アーキテクチャ上の制約
- **循環参照の禁止**: 2つのモジュールが互いに依存し合う状態を避け、依存の方向は常に「上位（Domain/Application）」から「下位（Infrastructure）」へ向かうようにする。
- **過剰なパターンの回避**: デザインパターンの適用は、あくまで「複雑さの解決」が目的である。パターンの導入によって逆に複雑性が増す場合は、シンプルな実装を選択する（YAGNI原則）。
