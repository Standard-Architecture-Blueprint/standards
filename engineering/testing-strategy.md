# Testing Strategy (テスト戦略規約)

## 1. 📌 概要
本ドキュメントは、複数のリポジトリ群において、バグの早期発見とデグレード（先祖返り）防止を実現するためのテスト方針を定義する。AP試験で扱われる「V字モデル」に基づき、各開発工程に対応したテストを実施する。

## 2. 🧪 テストピラミッド
効率的な品質向上のため、以下の比率を意識したテスト構成とする。

- **Unit Tests (単体テスト / 底辺)**:
  - 対象: クラス、メソッド単位のロジック。
  - 実行環境: ローカル、CI（DB等はモックを使用）。
  - 目標: 全テストコードの 70% 以上。高速に実行できること。
- **Integration Tests (結合テスト / 中位)**:
  - 対象: APIエンドポイント、DB連携、外部サービス連携。
  - 実行環境: コンテナ化された実際のDB（Testcontainers等）を使用。
  - 目標: 全テストコードの 20% 程度。
- **E2E Tests (システムテスト / 頂点)**:
  - 対象: SoEからSoRまでのエンドツーエンドの業務シナリオ。
  - 実行環境: 検証環境（Staging）。
  - 目標: 全テストコードの 10% 程度。クリティカルパスに絞る。

## 3. 📝 テスト手法の選定
AP試験の「テスト手法」に基づき、以下を使い分ける。

### 3.1. ホワイトボックステスト (主にUnit Test)
- **制御フローテスト**: コードの論理構造に基づき、命令網羅（C0）および判定条件網羅（C1）を確認する。
- **目標**: コアなビジネスロジックについては C1 網羅率 80% 以上を維持する。

### 3.2. ブラックボックステスト (主にIntegration/E2E Test)
- **同値分割・境界値分析**: 有効な入力範囲とその境界値、および無効な入力に対するエラーハンドリングを確認する。
- **原因結果グラフ**: 複雑な条件の組み合わせによる動作を整理し、テストケースを抽出する。

## 4. ⚙️ 自動テストの運用
- **CI連携**: プルリクエスト作成時にすべての Unit/Integration Test が自動実行され、1つでも失敗した場合はマージをブロックする。
- **回帰テスト (Regression Test)**: 修正による既存機能への影響がないことを確認するため、既存のテストセットを常に全件実行する。

## 5. 📊 品質指標 (Quality Metrics)
AP試験の「ソフトウェア品質」の観点から、以下のメトリクスを監視する。
- **テスト密度**: コード規模（ステップ数）に対するテストケース数の適切性。
- **バグ摘出密度**: テスト実施数に対する累積バグ発見数の推移（信頼度成長曲線）。
- **カバレッジ**: コードがテストによってどの程度実行されたか。

## 6. ⚠️ テストデータの取り扱い
- 本番データの利用は原則禁止とする。
- 機密情報（個人情報等）を含まない、テスト専用のダミーデータを生成して使用する。
