# GitHub Actions Templates (CI/CD Pipeline)

## 📌 概要
本プロジェクトは、13のサービスにおける品質管理とデプロイを自動化するワークフロー定義である。応用情報技術者試験（AP）の「ソフトウェア保守」および「構成管理」に基づき、信頼性の高いリリースサイクルを実現する。

## 📂 ディレクトリ構造と役割
ワークフローの重複を避け、再利用性を高める（Reusable Workflows）構成を厳守すること。
```text
github-actions/
├── workflows/          # 【CI/CD Pipeline】
│   ├── ci-backend.yml  # バックエンド用のビルド・テスト検品。
│   ├── ci-frontend.yml # フロントエンド用のビルド・テスト・Lint。
│   └── deploy.yml      # 承認ベースの環境デプロイ（CD）。
├── actions/            # 【Custom Actions】
│   ├── setup-java/     # Java環境とキャッシュのセットアップ共通処理。
│   └── setup-node/     # Node.js環境とキャッシュのセットアップ共通処理。
└── scripts/            # 【Automation Scripts】
    ├── version-up.sh   # セマンティックバージョニングの自動更新スクリプト。
    └── security-scan.sh# 脆弱性診断ツール（Trivy等）の実行スクリプト。
```
## 🛠️ 技術スタック（デファクトスタンダード）
AP試験の「システム開発環境」に基づき、業界標準のツールを選定。

| カテゴリ | 選定技術 | 理由・AP試験との関連 |
|:---|:---|:---|
| **CI Platform** | GitHub Actions | **継続的インテグレーション**のデファクト。 |
| **Artifact** | Docker | **コンテナ化**による環境の再現性と可搬性の確保。 |
| **SAST** | SonarCloud / Snyk | **静的解析**によるコードの脆弱性・品質の定量的評価。 |
| **Container Scan**| Trivy | イメージ内のライブラリ脆弱性チェック（セキュリティ対策）。 |

## 📐 実装・運用規約
AP試験の「ITサービスマネジメント」における変更管理の原則。

1. **品質ゲート（Quality Gate）の設置**: 
   - 全テストの通過、および静的解析の「パス」がなければマージを不可とする。
2. **秘匿情報の保護**: 
   - APIキーやパスワードは必ず `GitHub Secrets` を使用し、ログに露出させないこと。
3. **冪等性（Idempotency）の確保**: 
   - 何回実行しても、同じコードからは同じ結果（イメージ）が生成されるように構築すること。
4. **自動化の範囲**: 
   - 「ビルド→単体テスト→静的解析→イメージ作成」までは人の手を介さず自動実行する。

## 🚀 ワークフローの起動手順
1. **CIの実行**: 
   - `main` ブランチへのプルリクエスト作成時に自動起動。
2. **手動デプロイ**: 
   - `Actions` タブから対象ワークフローを選択し、デプロイ先環境を指定して実行。

## 🧪 品質管理
- **ワークフローの生存確認**: 
   - 定期実行（Scheduled Run）を行い、外部ライブラリのアップデート等によるビルド壊れを早期検知する。
