# 01_Requirement Definition (要件定義工程)

## 📌 概要
本ディレクトリでは、ビジネスニーズやユーザー要望を分析し、システムが備えるべき「機能」と「品質（非機能）」を定義する。AP試験における「要件定義プロセス」に準拠し、ステークホルダー間で合意された内容を成果物として集約する。

## 🏛️ 成果物と記述内容のマッピング
本工程で作成する成果物は、後の「システムテスト（ST）」および「受入テスト（UAT）」の合否判定基準となる。

| 成果物ファイル | 記述すべき具体的な内容 (AP試験準拠) | 作成単位 |
| :--- | :--- | :--- |
| **RD-000_index.md** | **機能一覧**、**用語集**、要件トレーサビリティマトリクス | 固定 (1ファイル) |
| **RD-001_common-req.md** | **非機能要件定義** (性能、可用性、セキュリティ、運用・保守性、移行要件) | 固定 (1ファイル) |
| **RD-{ID}_{feature}.md** | **業務フロー図 (As-Is/To-Be)**、**ユースケース図・記述**、制約事項 | 機能グループ単位 |

## 🛠️ 要件定義の主要アーティファクト

### 1. 業務フロー図 (Business Flow)
システム導入後の業務の流れを視覚化する。
- 役割（アクター）ごとのレーン分け
- データの発生源と最終的なアウトプットの明確化

### 2. ユースケース図・記述 (Use Case)
システムが提供する機能とアクターの相互作用を定義する。
- **基本フロー**: 正常系のシナリオ
- **代替フロー**: 条件分岐や例外処理のシナリオ

### 3. 非機能要件 (Non-Functional Requirements)
AP試験でも頻出の「品質特性」を漏れなく定義する。
- **可用性**: 稼働率、バックアップ頻度
- **性能**: レスポンス時間、スループット
- **セキュリティ**: 認証方式、アクセス制御、暗号化基準

## 🏷️ 命名・管理標準
- **ファイル名**: `RD-{ID}_{logic-name}.md`
- **要件ID**: 各要件には `REQ-xxx` 形式の固有IDを付与し、後の設計工程（FD/DD）へ継承させる。

## ✅ 完了定義 (Definition of Done)
- [ ] すべてのビジネスニーズが「機能」または「非機能」として定義されている。
- [ ] 用語の定義がステークホルダー間で一致している。
- [ ] 性能やセキュリティなどの非機能面が数値目標で示されている。
- [ ] システムテストのシナリオが作成可能なレベルまで具体化されている。
