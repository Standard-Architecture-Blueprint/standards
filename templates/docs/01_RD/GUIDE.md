# 📖 要件定義工程・作成手順書 (RD Writing Guide)

## 📌 1. 本工程のゴール
システムが解決すべき「ビジネス課題」を「機能・非機能要件」へと変換し、ステークホルダーと合意形成を行うこと。ここで定義された内容は、後の**システムテスト（ST）**の正解データとなる。

## ✍️ 2. 各成果物の作成手順

### 2.1 RD-000_index.md (機能一覧)
1. **要件IDの発行**: 新規機能に対し `REQ-100` 形式でIDを割り振る。
2. **優先度の設定**: AP試験の「優先順位付け」に基づき、必須か任意かを明確にする。
3. **用語集の更新**: 13リポジトリ間で解釈が分かれそうな言葉を逐次追記する。

### 2.2 RD-001_common-req.md (共通非機能要件)
1. **グレードの選択**: IPA「非機能要件グレード」を参考に、システムの重要度に応じた数値を埋める。
2. **制約の明記**: 13リポジトリ分割構成であること自体を「構造的制約」として記載する。

### 2.3 RD-{ID}_{feature-name}.md (個別要件)
1. **業務フローの描画**: 
   - 「誰が」「何をして」「どのリポジトリが反応するか」をスイムレーンで分ける。
2. **ユースケース記述の作成**:
   - **正常系**: ユーザーが目的を達成する最短ルートを書く。
   - **異常系**: バリデーションエラーやタイムアウト時の振る舞いを「代替フロー」に書く。

## 🔍 3. セルフチェックリスト (AP試験品質基準)
レビューに出す前に、作成者自身で以下の項目を確認すること。

- [ ] **完全性**: RD-000に記載した全ての機能に対し、個別RDファイルが存在するか？
- [ ] **一貫性**: RD-001で定めた応答速度などの基準と、個別RDの要求が矛盾していないか？
- [ ] **検証可能性**: その要件を見て、テストケースが作成可能か？（「使いやすいこと」等の曖昧な表現はないか）
- [ ] **追跡性**: 全ての機能要件に `REQ-xxx` のIDが付与されているか？

## 🔄 4. メンテナンスルール
- 要件が変更された場合、必ず `RD-000` の更新履歴を先に修正し、その後個別ファイルを更新する。
- 変更の影響範囲がどのリポジトリ（13個のうちどれか）に及ぶかを必ず明記する。
