# 6.1. Documentation Standards (ドキュメント管理・運用標準)

## 📌 概要
本ディレクトリは、SABプロジェクトにおけるドキュメントの「作成単位」「命名規則」「入出力の連鎖」を定義する。応用情報技術者試験（AP）の知識体系に基づき、各工程で作成すべき具体的な成果物を明確化し、複数のリポジトリ間で設計品質を平準化することを目的とする。

## 🏛️ 成果物ライフサイクルとV字モデル
AP試験の「共通フレーム（SLCP）」に基づき、各成果物は「設計の根拠（Input）」と「検証の基準（Target）」として機能する。

| 工程 | 作成・分割の単位 | 具体的な成果物要素 (何を書くか) | 対比するテスト (Target) |
| :--- | :--- | :--- | :--- |
| **01_RD (要件定義)** | **ビジネス機能単位** | 業務フロー図、ユースケース記述、機能一覧、非機能要件定義、用語集 | **ST** / **UAT** |
| **02_FD (外部設計)** | **I/F・リソース単位** | 画面遷移図、API仕様書、論理ER図、リポジトリ間連携仕様 | **IT** |
| **03_DD (内部設計)** | **モジュール・クラス単位** | クラス図、シーケンス図、物理DB設計、アルゴリズム詳細、状態遷移表 | **UT** |

## 📂 構成ファイル一覧と記述内容マッピング

### 01_RD (要件定義工程)
*   **RD-000_index.md**: [固定/1ファイル] 全要件の索引、進捗管理、**機能一覧**、**用語集**。
*   **RD-001_common-req.md**: [固定/1ファイル] **非機能要件定義**（性能、可用性、セキュリティ等）、システム全体制約。
*   **RD-{ID}_{feature-name}.md**: [可変/機能ごと] **業務フロー図**、**ユースケース図・記述**。

### 02_FD (外部設計工程)
*   **FD-000_common-spec.md**: [固定/1ファイル] 共通API規約、共通レスポンス形式、**外部システム接続仕様**。
*   **FD-001_data-model.md**: [固定/1ファイル] システム全体の**論理ER図**。RDのデータを構造化したもの。
*   **FD-{ID}_{feature-name}.md**: [可変/リソースごと] **API仕様書**（エンドポイント・I/O）、**画面遷移図**、**画面レイアウト**。

### 03_DD (内部設計工程)
*   **DD-000_architecture.md**: [固定/1リポジトリ1ファイル] **モジュール構造図**（パッケージ構成）、依存関係規定、利用フレームワーク詳細。
*   **DD-{ID}_{feature-name}.md**: [可変/コンポーネントごと] **クラス図**、**シーケンス図**、**物理DB設計**（インデックス・データ型）、**アルゴリズム詳細**、**状態遷移表**。

### 04_Test (テスト工程)
*   **TP-{LV}-{ID}_{target}.md**: [可変/試験単位] **テスト計画書**（検証範囲、テストデータ、環境、合否判定基準）。
*   **TR-{LV}-{ID}_{target}.md**: [可変/試験単位] **テスト報告書**（実施結果、バグ統計、エビデンス、リリース判定）。

## 🏷️ 命名・採番標準 (Naming Rules)
*   **形式**: `{工程/種別}-{レベル(任意)}-{ID}_{論理名}.md`
*   **{ID}**: 3桁の数字。同じビジネス機能には、RDからTRまで全工程で同じID（例：100）を割り当てる。
*   **{LV} (Test Level)**: `UT` (単体), `IT` (結合), `ST` (システム), `UAT` (受入)。

## 🛠️ トレーサビリティの実装
1.  **ID継承**: 下位工程（FD/DD）は必ず上位工程（RD）のIDをファイル名および項目名に含める。
2.  **監査タグ**: 各項目の末尾に `[Related: REQ-xxx]`（要件ID）等を記載する。これにより、「ソースコードのこの1行はどの要件に基づいているか」を逆方向に追跡可能（トレーサブル）にする。

## ✅ 更新・レビューフロー
*   **工程順守**: 上位工程（RD）の承認後に下位工程（FD/DD）を修正する（Forward-Only）。
*   **Atomic-Commit**: ドキュメント修正は、原則として対象のソースコードPRに含め、同時にマージする。
*   **Quality Gate**: レビュアーは「前工程の成果物（インプット）が漏れなく詳細化されているか」を主眼に確認する。
