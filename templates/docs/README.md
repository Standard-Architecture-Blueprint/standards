# 6.1. Documentation Standards (ドキュメント管理・運用標準)

## 📌 概要
本ディレクトリは、SABプロジェクトにおけるドキュメントの「作成単位」「命名規則」「管理フロー」を定義する。AP試験の「共通フレーム（SLCP）」に基づき、複数のリポジトリ間でドキュメントの整合性とトレーサビリティを担保することを目的とする。

## 🏛️ 成果物の作成単位 (Artifact Granularity)
AP試験の「システム開発ライフサイクル」に則り、下位工程に進むにつれて詳細化（分割）するデファクトスタンダードを採用する。

| 工程 | 作成・管理単位 | 決定の主眼 |
| :--- | :--- | :--- |
| **01_RD (要件定義)** | **ビジネス機能単位** | ユーザーが達成したい目的（例：受注管理、配送追跡） |
| **02_FD (外部設計)** | **インターフェース単位** | APIエンドポイント群、画面遷移ユニット、論理ER図 |
| **03_DD (内部設計)** | **モジュール単位** | クラス構成、コンポーネント、物理テーブル詳細 |
| **04_Test (テスト)** | **検証項目単位** | テストレベル（UT/IT/ST）ごとの計画と結果報告 |

## 🏷️ 命名規則と採番標準 (Naming Convention)
複数のリポジトリを横断して検索性を維持するため、**「工程コード-機能ID_論理名.md」**の形式を強制する。

- **工程コード**: `RD`, `FD`, `DD`, `TP`(計画), `TR`(報告)
- **機能ID**: 3桁の数字。リポジトリ間でも同じビジネス機能には同じIDを割り当てる（例：認証系は100系）。
- **論理名**: ケバブケース（kebab-case）で、そのファイルの内容を明示する。

> **例:**
> - `RD-100_auth-process.md` (認証ビジネス要件)
> - `FD-100_auth-api-spec.md` (認証API仕様)
> - `DD-100_auth-logic.md` (認証内部ロジック)

## 📂 ディレクトリ構造

- **[01_RD/](./01_RD/)**: 要件定義。ビジネスプロセスと機能・非機能要件の定義。
- **[02_FD/](./02_FD/)**: 外部設計。サブシステム間の境界とユーザーインターフェースの確定。
- **[03_DD/](./03_DD/)**: 内部設計。実装者がコード化できるレベルへの詳細化。
- **[04_Test/](./04_Test/)**: テスト。検証計画と品質エビデンスの記録。
- **[99_Management/](./99_Management/)**: ADR（意思決定記録）や運用マニュアルの管理。

## 🛠️ トレーサビリティの運用 (Traceability)
AP試験の「システム監査」に対応するため、ドキュメントを跨いだIDの紐付けを必須とする。

1.  **要件ID**: `RD` 内で `REQ-xxx` を定義する。
2.  **設計への紐付け**: `FD`/`DD` の各項目に `[Related: REQ-xxx]` を付与。
3.  **テストへの紐付け**: テストケースに `[Target: REQ-xxx]` を付与。

## ✅ ドキュメント更新・レビューフロー
1.  **工程順守**: 上位工程（RD）の修正を承認してから、下位工程（FD/DD）を修正する。
2.  **同期マージ**: ドキュメントの修正は、原則としてソースコードのプルリクエスト（PR）に含め、同時にマージする。
3.  **整合性確認**: レビュアーは、要件の変更が設計・テストまで正しく伝播しているかを確認する。
