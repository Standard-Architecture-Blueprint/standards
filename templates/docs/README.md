# ドキュメント・テンプレート (Documentation Templates)

## 📌 概要
本ディレクトリは、システム開発の各フェーズ（要件定義、外部設計、内部設計）における標準成果物の雛形を定義する。AP試験の「共通フレーム（SLCP）」に基づき、工程ごとの入力と出力を明確化し、複数のリポジトリ間でドキュメントの品質と粒度を統一することを目的とする。

## 🏛️ ドキュメント体系 (Standard Hierarchy)

AP試験の「システム開発工程」に従い、以下のディレクトリ構造でドキュメントを管理する。

### [01_RD/ (要件定義工程)](./01_RD/)
**利害関係者要件をシステム仕様へ翻訳する**
- **RD-001_Business.md**: 業務フロー、ビジネスルール、対象範囲（Scope）。
- **RD-002_System.md**: 機能要件、非機能要件（可用性、性能、セキュリティ、運用性）。
- **目的**: 「何を作るか」を確定させ、後続工程との合意形成を図る。

### [02_FD/ (外部設計工程)](./02_FD/)
**ユーザー視点およびサブシステム間のインターフェースを定義する**
- **FD-001_UI-UX.md**: 画面遷移図、レイアウト、入出力バリデーション。
- **FD-002_API.md**: APIエンドポイント、リクエスト/レスポンス、エラー定義。
- **FD-003_Data.md**: 概念/論理データモデル、ER図、コード定義。
- **目的**: ユーザーおよび他サービスとの「境界」を確定させる。

### [03_DD/ (内部設計工程)](./03_DD/)
**プログラムの実装単位まで構造を具体化する**
- **DD-001_Module.md**: クラス構成、コンポーネント分割、依存関係。
- **DD-002_Sequence.md**: オブジェクト間の動的振る舞い（シーケンス）。
- **DD-003_Physical.md**: DB物理設計（インデックス、型指定、パーティション）。
- **目的**: 実装者が「どう作るか」に迷わない詳細度まで詳細化する。

## 📐 記述の基本原則 (Documentation Principles)
AP試験の「品質管理」に基づき、以下の原則を遵守せよ。

- **トレーサビリティの維持**: 全ての設計項目は、上位の要件（RD）のIDと紐付かなければならない。
- **一貫性と整合性**: 外部設計（FD）で定義したデータ項目が、内部設計（DD）やコードと乖離していないことを確認する。
- **最新性の保持**: コードの変更に合わせてドキュメントも更新し、常に「生きた仕様書」として維持する。
- **図解の活用**: Mermaid 等を利用し、テキストだけでは伝わりにくい構造（クラス、シーケンス、ER）を視覚化する。

## ✅ ドキュメント・レビュー合否基準
成果物は、以下のチェックリストをパスしなければならない。

- [ ] 非機能要件（特にセキュリティと性能）が具体的に定義されているか。
- [ ] 例外系・準正常系の処理フローが明記されているか。
- [ ] サブシステム間のデータの受け渡しルール（プロトコル、形式）が明確か。
- [ ] 専門用語が定義され、プロジェクト内で表記揺れがないか。
