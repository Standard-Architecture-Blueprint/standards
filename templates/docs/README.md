# 6.1. Documentation Standards (ドキュメント管理・運用標準)

## 📌 概要
本ディレクトリは、SABプロジェクトにおけるドキュメントの「作成単位」「命名規則」「管理フロー」を定義する。AP試験の「共通フレーム（SLCP）」に基づき、複数のリポジトリ間でドキュメントの整合性とトレーサビリティを担保することを目的とする。

## 🏛️ 成果物の作成単位と入出力の連鎖 (Lifecycle & Traceability)
各工程は前工程の成果物を「入力（根拠）」として受け取り、詳細化した「出力（設計）」を次工程へ渡す。この連鎖（バトンパス）を厳守することで、品質と監査証跡を確保する。

| 工程 | 作成単位 | 前工程からの入力 (Input) | 本工程の出力 (Output) | 次工程/テストへの役割 |
| :--- | :--- | :--- | :--- | :--- |
| **01_RD (要件定義)** | **ビジネス機能** | 業務課題・ユーザー要望 | **要件定義書** (機能・非機能要件) | システム全体の振る舞いの基準 / ST・UATの期待値 |
| **02_FD (外部設計)** | **I/F・サブシステム** | 要件定義の機能一覧・制約 | **方式設計書** (API・画面・論理ER) | コンポーネント間の境界条件 / ITの期待値 |
| **03_DD (内部設計)** | **モジュール** | 外部設計のI/O・データ定義 | **詳細設計書** (クラス・シーケンス・物理DB) | プログラムの構造・製造指示書 / UTの期待値 |
| **04_Test (テスト)** | **検証項目** | 各設計書の期待値・テスト計画 | **テスト報告書** (品質エビデンス) | リリース可否の最終判断根拠 |



## 🏷️ 命名規則と採番標準 (Naming Rules)
複数のリポジトリを横断して検索性を維持するため、以下の形式を強制する。

### 基本設計・詳細設計
`{工程}-{ID}_{feature-name}.md`
- **工程コード**: `RD`, `FD`, `DD`
- **ID**: 3桁の数字。リポジトリ間でも同じビジネス機能には同じIDを割り当てる（例：認証系は100系）。

### テストドキュメント (V字モデル対応)
`{種別}-{レベル}-{ID}_{target-name}.md`
- **種別**: `TP` (計画), `TR` (報告)
- **レベル**: `UT` (単体), `IT` (結合), `ST` (システム), `UAT` (受入)
- **命名例**: `TP-IT-100_auth-api.md` (ID:100 認証APIの結合テスト計画)

## 📝 構成ファイル一覧

### 01_RD (要件定義工程)
*   **RD-000_index.md**: [固定] 全要件の索引および進捗管理。全工程のIDの起点。
*   **RD-001_common-req.md**: [固定] 共通要件・非機能要件（性能・セキュリティ等）。
*   **RD-{ID}_{feature-name}.md**: [可変] 業務フロー、ユースケース定義。

### 02_FD (外部設計工程)
*   **FD-000_common-spec.md**: [固定] 共通I/O、レスポンス形式、API規約。
*   **FD-001_data-model.md**: [固定] 論理ER図。RDのエンティティを構造化したもの。
*   **FD-{ID}_{feature-name}.md**: [可変] API定義、画面遷移、リポジトリ間連携仕様。

### 03_DD (内部設計工程)
*   **DD-000_architecture.md**: [固定] パッケージ構造、依存関係規定。
*   **DD-{ID}_{feature-name}.md**: [可変] クラス設計、シーケンス、物理DB（インデックス等）。

### 04_Test (テスト工程)
*   **TP-{LV}-{ID}_{target}.md**: [可変] 各レベルのテスト計画。検証範囲と環境の定義。
*   **TR-{LV}-{ID}_{target}.md**: [可変] 各レベルのテスト報告。RD/FD/DDに対する適合性証明。

### 99_Management (管理・運用)
*   **ADR-{ID}_{decision-title}.md**: [可変] アーキテクチャ決定記録。選定理由とトレードオフ。
*   **Manual-{ID}_{target}.md**: [可変] 運用・操作マニュアル。

## 🛠️ トレーサビリティの実装 (Internal Link)
1. **ID継承**: 下位工程は必ず上位工程のIDを継承したファイル名・項目名にする。
2. **監査用タグ**: 各設計書の項目末尾に `[Related: REQ-xxx]`（要件ID）等を記載し、双方向に辿れるようにする。

## ✅ 更新・レビューフロー
*   **Forward-Only**: 上位工程（RD）の修正・承認をトリガーに、順次下位工程を更新する。
*   **Atomic-Commit**: ドキュメント修正は、原則として対象のソースコードPRに含める。
*   **Quality Gate**: レビュアーは「前工程の制約が漏れなく反映されているか」を主眼に確認する。
