# 6.1. Documentation Standards (ドキュメント管理・運用標準)

## 📌 概要
本ディレクトリは、SABプロジェクトにおけるドキュメントの「作成単位」「命名規則」「管理フロー」を定義する。AP試験の「共通フレーム（SLCP）」に基づき、複数のリポジトリ間でドキュメントの整合性とトレーサビリティを担保することを目的とする。

## 🏛️ 成果物の作成単位 (Artifact Granularity)
下位工程に進むにつれて詳細化（分割）するデファクトスタンダードを採用する。

| 工程 | 作成・管理単位 | 決定の主眼 |
| :--- | :--- | :--- |
| **01_RD (要件定義)** | **ビジネス機能単位** | ユーザーが達成したい目的（例：受注管理） |
| **02_FD (外部設計)** | **インターフェース単位** | API定義、画面遷移、論理ER図 |
| **03_DD (内部設計)** | **モジュール単位** | クラス構成、コンポーネント、物理DB詳細 |
| **04_Test (テスト)** | **検証項目単位** | テストレベル別の計画と報告 |

## 🏷️ テストドキュメント命名規則 (Test Naming Convention)
AP試験の「V字モデル」に基づき、「書類種別」「テストレベル」「対応ID」を組み合わせた命名を行う。

### 基本形式
`{種別}-{レベル}-{ID}_{target-name}.md`

### 1. 種別コード (Document Type)
*   **TP**: Test Plan（テスト計画書）
*   **TR**: Test Report（テスト報告書）

### 2. テストレベルコード (Test Level)
AP試験の用語に準拠し、検証対象に合わせて選択する。
*   **UT**: Unit Test（単体テスト / 内部設計 DD に対応）
*   **IT**: Integration Test（結合テスト / 外部設計 FD に対応）
*   **ST**: System Test（システムテスト / 要件定義 RD に対応）
*   **UAT**: User Acceptance Test（受入テスト / 業務要件に対応）

### 3. 命名例
*   `TP-IT-100_auth-api.md`: ID:100 認証APIの**結合テスト計画**
*   `TR-ST-100_auth-function.md`: ID:100 認証機能の**システムテスト報告**
*   `TR-UT-200_payment-logic.md`: ID:200 決済ロジックの**単体テスト報告**（書面化が必要な場合）

## 📝 構成ファイル一覧

### 01_RD (要件定義工程)
*   **RD-000_index.md**: [固定] 全要件の索引および進捗管理表。
*   **RD-001_common-req.md**: [固定] システム共通要件および非機能要件の定義。
*   **RD-{ID}_{feature-name}.md**: [可変] 個別ビジネス機能ごとの要件定義。

### 02_FD (外部設計工程)
*   **FD-000_common-spec.md**: [固定] 共通I/O、レスポンス形式、API規約。
*   **FD-001_data-model.md**: [固定] 論理ER図および共通エンティティの定義。
*   **FD-{ID}_{feature-name}.md**: [可変] 個別機能のI/O設計およびデータモデル詳細。

### 03_DD (内部設計工程)
*   **DD-000_architecture.md**: [固定] パッケージ構造、依存関係、フレームワーク利用規定。
*   **DD-{ID}_{feature-name}.md**: [可変] 個別機能の内部ロジックおよびシーケンス詳細。

### 04_Test (テスト工程)
*   **TP-{LV}-{ID}_{target}.md**: [可変] 各レベルのテスト計画書。
*   **TR-{LV}-{ID}_{target}.md**: [可変] 各レベルのテスト報告書。

### 99_Management (管理・運用)
*   **ADR-{ID}_{decision-title}.md**: [可変] アーキテクチャ決定記録。
*   **Manual-{ID}_{target}.md**: [可変] 運用・操作マニュアル。

## 🛠️ トレーサビリティの運用 (Traceability)
AP試験の「システム監査」に対応するため、ドキュメントを跨いだIDの紐付けを必須とする。
1. **要件ID**: `RD` 内で `REQ-xxx` を定義する。
2. **設計・テストへの紐付け**: `FD`/`DD`/`Test` の各項目に `[Related: REQ-xxx]` を付与。

## ✅ 更新・レビューフロー
*   **工程順守**: 上位工程（RD）の修正を承認してから、下位工程（FD/DD）を修正する。
*   **同期マージ**: ドキュメントの修正は、原則としてソースコードのPRに含め、同時にマージする。
