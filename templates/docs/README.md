# 6.1. Documentation Standards & Templates (ドキュメント管理標準)

## 📌 概要
本ディレクトリは、SABプロジェクトにおけるドキュメントの「管理単位」「命名規則」「雛形」を定める。AP試験の「構成管理」に基づき、複数のリポジトリ間でドキュメントの検索性、整合性、トレーサビリティを最大化することを目的とする。

## 📂 ドキュメントの管理単位
ドキュメントは、原則として**「機能（Feature）またはサブシステム」**単位で分割して作成する。
巨大な1ファイル化を避け、以下の単位でファイルを分割することで、変更時のコンフリクトを防止し、変更履歴を明確にする。

1.  **共通定義**: システム全体に跨る仕様（データモデル、共通API規約等）。
2.  **機能単位**: ユーザーが操作する最小単位（ログイン、注文登録、履歴照会等）。
3.  **非機能単位**: セキュリティ、バックアップ、バッチ実行等のインフラ・運用仕様。

## 🏷️ 命名規則と採番標準
ファイル名は、**「工程コード-連番_論理名.md」**の形式を標準とする。

- **工程コード**: `RD`(要件), `FD`(外部設計), `DD`(内部設計), `TP`(テスト計画), `TR`(テスト報告)
- **連番**: 3桁の数字。`000`は索引、`1xx`は機能群A、`2xx`は機能群Bのように割り当てる。
- **論理名**: ケバブケース（kebab-case）を使用。

> **例:**
> - `RD-100_user-auth.md` (ユーザー認証の要件定義)
> - `FD-100_user-auth.md` (ユーザー認証の画面・API設計)
> - `DD-100_user-auth.md` (ユーザー認証の詳細設計)

## 🏛️ ドキュメント体系

### [01_RD/ (要件定義工程)](./01_RD/)
- **RD-000_index.md**: 全要件のトレーサビリティマップ（要件ID一覧）。
- **RD-1xx_{feature}.md**: 個別機能の要件（ビジネスルール、入出力要件）。

### [02_FD/ (外部設計工程)](./02_FD/)
- **FD-000_common-spec.md**: 共通レスポンス形式、共通バリデーション規約。
- **FD-1xx_{feature}.md**: 個別機能のI/O設計、論理データモデル（ER図）。

### [03_DD/ (内部設計工程)](./03_DD/)
- **DD-1xx_{feature}.md**: 実装ロジック、シーケンス図、物理テーブル定義。

### [04_Test/ (テスト工程)](./04_Test/)
- **TP-xxx_{target}.md**: テスト計画。範囲、環境、合格基準。
- **TR-xxx_{target}.md**: テスト報告。実施結果、バグ収束状況、リリース判定。

### [99_Management/ (管理・運用)](./99_Management/)
- **ADR-xxx_{title}.md**: アーキテクチャ決定記録。選定根拠とトレードオフ。

## 🛠️ トレーサビリティの運用
AP試験の「システム監査」に耐えうる証跡管理のため、以下のID管理を徹底する。

1.  **要件IDの定義**: `RD` 内の各項目に `REQ-001` 等の不変IDを付与する。
2.  **設計・テストへの伝播**: 
    - `FD`/`DD` の各章末に `[Related: REQ-001]` と記載。
    - テストケースの前提条件に `[Target: REQ-001]` と記載。

## ✅ 更新ワークフロー
1.  **新規作成**: テンプレートをコピーし、未使用の連番を採番して作成。
2.  **影響修正**: `RD` を修正した場合、紐付く ID を辿り `FD`/`DD`/`Test` をすべて更新する。
3.  **PR検証**: コードの変更とドキュメントの更新が対になっているかをマージ条件とする。
