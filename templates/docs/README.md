# ドキュメント・標準テンプレート (Documentation Templates)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) におけるシステム開発ライフサイクル（SDLC）の各工程で作成すべき成果物の雛形を定義する。AP試験の「共通フレーム（SLCP）」および「システム開発工程」に基づき、13のリポジトリ間でドキュメントの品質・粒度・トレーサビリティを統一することを目的とする。

## 🏛️ ドキュメント体系と工程定義

AP試験の「ソフトウェア開発プロセス」に従い、以下のディレクトリ構造でドキュメントを管理・運用する。

### [01_RD/ (要件定義工程)](./01_RD/)
**利害関係者のニーズをシステム仕様に翻訳し、合意形成を行う。**
- **RD-001_Business.md**: 業務要件。ビジネスフロー、ルール、対象範囲の定義。
- **RD-002_System.md**: 機能要件・非機能要件（可用性、性能、セキュリティ、保守・運用性）。

### [02_FD/ (外部設計工程)](./02_FD/)
**ユーザー視点の仕様およびサブシステム間のインターフェースを確定させる。**
- **FD-001_UI-UX.md**: 画面遷移図、レイアウト設計、入出力バリデーション。
- **FD-002_API.md**: API定義。プロトコル、エンドポイント、データ形式。
- **FD-003_Data.md**: 論理データ設計。ER図、エンティティ定義、正規化。

### [03_DD/ (内部設計工程)](./03_DD/)
**実装者が物理的制約を考慮してコード化できるレベルまで詳細化する。**
- **DD-001_Module.md**: 構造設計。クラス構成、コンポーネント分割、依存関係。
- **DD-002_Sequence.md**: 動的設計。オブジェクト間のメッセージパッシング（シーケンス図）。
- **DD-003_Physical.md**: 物理データ設計。インデックス、型、パーティショニング、整合性制約。

### [04_Test/ (テスト工程)](./04_Test/)
**設計通りに動作することを検証し、品質を客観的なエビデンスで証明する。**
- **Test-Plan.md**: テスト計画。範囲、環境、スケジュール、合格基準の定義。
- **Test-Report.md**: テスト報告。実施結果、バグ収束状況、品質分析、リリース判定。

### [99_Management/ (管理・運用)](./99_Management/)
**プロジェクトの意思決定と、リリース後の安定稼働を支える情報を管理する。**
- **ADR-Template.md**: アーキテクチャ決定記録。技術選定の論理的根拠とトレードオフ。
- **Manual-Template.md**: 運用・操作マニュアル。起動停止、リカバリ手順、操作方法。

## 📐 記述の基本原則 (Documentation Principles)
AP試験の「品質管理」および「システム監査」に基づき、以下の原則を遵守せよ。

1.  **トレーサビリティ (Traceability)**: 全ての設計項目は、上位の要件IDと紐付け、漏れや余計な実装がないことを保証する。
2.  **不変性の確保 (Version Control)**: ドキュメントは Git で管理し、いつ、誰が、なぜ変更したかを明確にする。
3.  **図解の標準化**: クラス、シーケンス、ER図などは Mermaid 等を用い、コードと同期しやすい形式で視覚化する。
4.  **最新性の維持**: 仕様変更時は「コードを直してからドキュメントを直す」のではなく、設計変更を承認してから実装を行うプロセスを遵守する。

## ✅ 成果物の承認基準 (Quality Gate for Docs)
ドキュメント・レビューにおいて、以下のチェックリストをパスしなければならない。

- [ ] 非機能要件（可用性・セキュリティ等）の具体的な目標値が設定されているか。
- [ ] 境界値や異常系（エラーハンドリング）の挙動が明記されているか。
- [ ] 専門用語が定義され、プロジェクト内での表記揺れがないか。
- [ ] 監査証跡として利用可能なレベルで、判断の根拠（ADR等）が残されているか。
