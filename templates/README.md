# 6. Project Templates (プロジェクトテンプレート)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) プロジェクトにおける全ての成果物（ドキュメント・コード）の「型」を定める。AP試験の「ソフトウェア再利用」および「共通フレーム（SLCP）」に基づき、13のリポジトリ間で品質のばらつきを排除し、開発の着手スピードを最大化することを目的とする。

## 🏛️ 管理資産（テンプレート一覧）

### [docs/](./docs/)
**RD/FD/DDのMarkdown雛形**
- **要件定義 (RD)**: 利害関係者要件、機能・非機能要件の定義形式。
- **外部設計 (FD)**: 画面遷移、APIインターフェース、バッチスケジュールの定義形式。
- **内部設計 (DD)**: クラス図、シーケンス図、DB物理設計の定義形式。
- **目的**: AP試験の「システム開発工程」における各フェーズの成果物を標準化し、ドキュメントの網羅性を担保する。

### [code/](./code/)
**各言語リポジトリの初期構造**
- **ディレクトリ構造**: `src/`, `tests/`, `config/`, `infra/` の標準的なディレクトリ配置。
- **ボイラープレート**: 各言語（Java/Spring Boot等）における基底クラスや、共通ライブラリの読み込み設定。
- **目的**: 「モジュール分割」の原則を初期段階から強制し、リポジトリごとの構造的負債を未然に防ぐ。

## 🛠️ テンプレート活用の原則
AP試験の「開発環境の標準化」に基づき、以下の運用を遵守せよ。

1.  **テンプレート起点の徹底**: 新規リポジトリ作成や、新しい設計工程に着手する際は、必ず本ディレクトリの最新テンプレートをクローンして使用すること。
2.  **トレーサビリティの確保**: `docs/` 内の各ドキュメントは、要件ID（RD）から詳細設計（DD）までを一貫して追跡できるリンク構造を持たせること。
3.  **継続的改善（フィードバック）**: 開発過程で見つかったテンプレートの不備や、より効率的な構造は、本ディレクトリへ速やかにフィードバックし、全リポジトリへ展開すること。

## ✅ テンプレート導入のメリット
AP試験の「プロジェクトマネジメント」指標に照らし、以下の効果を測定する。

- **品質の均一化**: どのリポジトリ、どの設計書を見ても、同一のレベルで情報が整理されている状態を作る。
- **レビュー工数の削減**: 構造が標準化されているため、レビュアーが「内容の妥当性」という本質的評価に集中できる。
- **ナレッジの共有**: 優れた設計・実装パターンを「型」として共有することで、チーム全体の技術レベルを底上げする。
