# Security Design Guidelines (セキュリティ設計指針)

## 1. 📌 概要
本ドキュメントは、システムの機密性(C)、完全性(I)、可用性(A)を維持するためのセキュリティ基準を定義する。特定の境界（ネットワーク）に依存せず、すべてのアクセスを検証する「ゼロトラスト」の考え方を基本原則とする。

## 2. 🛡️ 防御の基本原則
### 2.1. 最小権限の原則 (Least Privilege)
- ユーザ、プロセス、およびシステムコンポーネントには、その責務を果たすために必要な最小限の権限のみを付与する。

### 2.2. 多層防御 (Defense in Depth)
- ネットワーク、OS、アプリケーション、データの各レイヤで個別のセキュリティ対策を講じ、単一の防御破綻がシステム全体の崩壊に繋がらないようにする。

### 2.3. 入力の検証 (Input Validation)
- すべての外部入力（APIリクエスト、ファイル、ヘッダー）は悪意があるものと見なし、バックエンド（SoR）で厳格なバリデーションを行う。

## 3. 🔐 認証と認可 (Authentication & Authorization)
### 3.1. 認証 (Who are you?)
- **方式**: OpenID Connect (OIDC) に基づくトークン認証。
- **パスワード管理**: システム側で平文のパスワードを保持しない。ハッシュ化（PBKDF2, Argon2等）およびソルトを使用する。
- **多要素認証 (MFA)**: セキュリティ要求の高い機能や管理者権限の実行には、追加の認証を要求する。

### 3.2. 認可 (What can you do?)
- **方式**: ロールベースアクセス制御 (RBAC)。
- **実装**: JWT の `claims` に含まれるロール情報に基づき、API エンドポイントおよびビジネスロジックレベルでアクセスを制御する。

## 4. 🚀 主要な脆弱性への対策
OWASP Top 10 を基準とし、特に以下の攻撃を防止する。

| 攻撃種別 | 対策方針 |
|:---|:---|
| **SQLインジェクション** | プレースホルダ（静的プリペアードステートメント）を原則とし、動的SQLを禁止する。 |
| **XSS (Cross-Site Scripting)** | SoEレイヤでの出力エスケープを徹底し、Content Security Policy (CSP) を設定する。 |
| **CSRF** | ステートレスなJWT認証を採用し、Cookieベースのセッション管理を避ける。 |
| **OSコマンドインジェクション** | 外部入力をシェルコマンドの引数として直接渡さない。 |

## 5. 💎 データの保護
### 5.1. 通信の暗号化 (In-Transit)
- 全ての通信経路（SoE-Platform、Platform-SoR）において TLS 1.3 以降を強制する。

### 5.2. 保存データの暗号化 (At-Rest)
- データベースのディスク暗号化を行い、機密性の高い個人情報（PII）はカラムレベルでの暗号化を検討する。

## 6. 📝 監査ログと監視
- **証跡管理**: 「いつ、誰が、どのデータに、何をしたか」を特定できる監査ログを保存する。
- **ログの保護**: 監査ログは改ざん不能なストレージに保存し、一般の開発者・運用者が削除できない権限設定とする。
- **異常検知**: 短時間の大規模アクセスや連続したログイン失敗を検知し、アラートを通知する。
