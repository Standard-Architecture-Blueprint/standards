# Data Modeling Standards (データ設計標準規約)

## 1. 📌 概要
本ドキュメントは、システム全体で一貫したデータ構造を維持するための設計指針を定義する。データの整合性、再利用性、および将来的なデータ分析（SoIレイヤへの活用）を考慮したモデリングを行う。

## 2. 📐 モデリング原則
### 2.1. 正規化 (Normalization)
- **原則**: データの冗長性を排除し、更新異常を防ぐため **第3正規形** まで正規化を行う。
- **例外**: パフォーマンス上の理由で非正規化を行う場合は、ADRにてその理由と整合性維持の手法を明記すること。

### 2.2. 主キー (Primary Key) 設計
- **サロゲートキーの推奨**: 業務的な意味を持たない連番（BIGINT / Identity）または UUID を主キーとする。
- **ナチュラルキーの回避**: メールアドレスや電話番号など、変更の可能性がある属性を主キーにしない。

### 2.3. 制約の活用
- **参照整合性制約 (Foreign Key)**: テーブル間のリレーションには必ず外部キー制約を付与する。
- **NOT NULL 制約**: 可能な限り NULL を許容しない設計とし、デフォルト値を検討する。

## 3. 📝 命名規則
一貫性を保つため、以下のスネークケース（snake_case）による命名を統一する。

| 対象 | 命名規則 | 例 |
|:---|:---|:---|
| テーブル名 | 複数形の名詞 | `users`, `order_details` |
| カラム名 | 単数形の名詞 | `user_id`, `first_name` |
| 主キー | `id` または `{table_name_singular}_id` | `id` |
| 外部キー | `{参照先テーブル名_singular}_id` | `user_id` |
| インデックス | `idx_{table}_{column}` | `idx_users_email` |
| 一意制約 | `uk_{table}_{column}` | `uk_users_email` |

## 4. 📋 共通カラム (監査用カラム)
すべてのテーブルにおいて、追跡可能性（Traceability）を確保するために以下のカラムを必須とする。

| カラム名 | 型 | 説明 |
|:---|:---|:---|
| `created_at` | TIMESTAMP | レコード作成日時 |
| `updated_at` | TIMESTAMP | レコード最終更新日時 |
| `created_by` | VARCHAR | 作成者ID（システムユーザーID等） |
| `updated_by` | VARCHAR | 更新者ID |
| `version` | INTEGER | 楽観的ロック用のバージョン番号 |
| `is_deleted` | BOOLEAN | 論理削除フラグ（必要に応じて採用） |

## 5. 🚀 パフォーマンス設計
### 5.1. インデックス (Index)
- 外鍵、検索条件に頻繁に利用されるカラム、ソート対象のカラムには適切にインデックスを配置する。
- 複合インデックスを貼る際は、カラムの順序（カーディナリティの高い順）に留意する。

### 5.2. データ型
- データの性質に合わせた最適な型を選択する（例: 金額は誤差を防ぐため `DECIMAL` を使用し `FLOAT` は避ける）。
- 文字列型は必要最小限の長さを指定し、安易に `TEXT` 型を多用しない。
