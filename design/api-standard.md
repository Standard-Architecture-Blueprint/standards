# インターフェース設計規約 (API Standard)

## 📌 概要
本ドキュメントは、各サービス間およびフロントエンド・バックエンド間の通信インターフェース規約を定める。AP試験の「ネットワークプロトコル」および「標準化」の原則に基づき、相互運用性と保守性の高いAPIを構築することを目的とする。

## 1. RESTful API 基本規約
原則として、資源（リソース）中心の設計を行い、HTTPメソッドを適切に使い分ける。

### 1.1. エンドポイント設計
- **URL構成**: `https://api.example.com/[service-name]/v1/[resources]`
- **リソース名**: 複数形の名詞を使用する（例: `/users`, `/orders`）。
- **小文字統一**: パスには小文字を使用し、単語区切りはハイフン（ケバブケース）を用いる。

### 1.2. HTTPメソッドの利用
AP試験の「HTTPプロトコル」の理解に基づき、以下の意味論を厳守する。

| メソッド | 用途 | べき等性 | 備考 |
| :--- | :--- | :--- | :--- |
| **GET** | リソースの取得 | あり | 副作用（データ更新）禁止。 |
| **POST** | リソースの作成 | なし | 実行のたびに新しいリソースが生成される。 |
| **PUT** | リソースの完全更新 | あり | 指定した内容でリソースを置換する。 |
| **PATCH** | リソースの部分更新 | なし/あり | 差分のみを更新する。 |
| **DELETE** | リソースの削除 | あり | 削除済みリソースへの実行も成功扱いとする。 |

## 2. 共通レスポンス構造 (Common Response)
フロントエンドでの処理を共通化するため、レスポンスのエンベロープを統一する。
```json
{
  "status": "success",
  "data": { ... },
  "error": null,
  "timestamp": "2026-05-04T14:41:00Z"
}
```

#### HTTPステータスコードの定義
AP試験およびデファクトスタンダードに基づき、以下のコードを適切に使い分ける。

* **200 OK**: リクエスト成功。データの取得・更新が正常に完了した。
* **201 Created**: リソースの作成成功。POSTリクエストの結果として使用する。
* **204 No Content**: 成功したが返すべきデータがない（DELETE成功時など）。
* **400 Bad Request**: クライアント側の入力不備（バリデーションエラー）。
* **401 Unauthorized**: 認証が必要、または認証に失敗した（トークン無効など）。
* **403 Forbidden**: 認証はされているが、対象リソースへのアクセス権限がない。
* **404 Not Found**: 指定されたリソース（URLまたはID）が存在しない。
* **405 Method Not Allowed**: 許可されていないHTTPメソッドの使用。
* **409 Conflict**: リソースの状態と競合が発生（例：ユーザー名の重複登録）。
* **429 Too Many Requests**: レートリミット（短時間での過剰なリクエスト）超過。
* **500 Internal Server Error**: サーバー側の予期せぬエラー（バグ、DB停止など）。
* **503 Service Unavailable**: サーバーが一時的に過負荷、またはメンテナンス中。

## 3. gRPC 通信規約
13のリポジトリ間（内部通信）では、型安全性を担保するため gRPC を優先的に使用する。
- **IDL管理**: Protocol Buffers (.proto) を common-library または専用リポジトリで一元管理する。
- **破壊的変更の禁止**: フィールド番号の変更や削除は行わず、非推奨（deprecated）として扱う。

## 4. セキュリティ・認証 (Security & Auth)
AP試験の「ネットワークセキュリティ」に基づき、以下の対策を講じる。
- **認証**: JWT (JSON Web Token) を Authorization ヘッダー（Bearer）に付与する。
- **認可**: OAuth 2.0 / RBAC（ロールベースアクセス制御）に基づき、リソースへのアクセス可否を判定する。
- **暗号化**: すべての通信は TLS 1.3 以上で暗号化（HTTPS）する。
- **バリデーション**: すべての入力値に対して、サーバー側で厳格な型・範囲チェック（サニタイジング）を行う。

## 5. バージョニング方針
- **破壊的変更**: 既存クライアントが動作しなくなる変更を導入する場合は、URLパスのメジャーバージョン（例: /v1/ -> /v2/）を更新する。
- **マイナーアップデート**: フィールド追加などは後方互換性を維持する。
