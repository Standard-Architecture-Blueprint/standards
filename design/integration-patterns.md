# Integration Patterns (システム連携パターン規約)

## 1. 📌 概要
本ドキュメントは、マイクロサービス群および外部システム間での通信方式を定義する。データの即時性とシステムの可用性のバランスを最適化するための「連携の型」を標準化する。

## 2. 🔄 連携方式の分類
システム要求に応じて、以下の2種類の通信方式を使い分ける。

### 2.1. 同期連携 (Synchronous)
- **方式**: REST API (HTTP/HTTPS)
- **ユースケース**: 処理結果を即座に利用者に返す必要がある場合（照会処理、即時バリデーションが必要な登録等）。
- **メリット**: 実装が単純で、データの整合性を確認しやすい。
- **デメリット**: 呼び出し先のサービスがダウンしていると自サービスも影響を受ける（障害伝播）。

### 2.2. 非同期連携 (Asynchronous)
- **方式**: メッセージキュー、イベントバス、バッチ連携
- **ユースケース**: 処理に時間がかかる場合や、相手の応答を待たずに自処理を完結させたい場合（メール送信、大量データ加工、ログ収集等）。
- **メリット**: システム間の結合度が低く（疎結合）、負荷分散や耐障害性に優れる。
- **デメリット**: 結果整合性（Eventual Consistency）の許容が必要。

## 3. 🛠️ 共通連携パターン
### 3.1. API ゲートウェイ・パターン
- **内容**: 外部（SoE）からのリクエストを一度 Platform レイヤのゲートウェイで受け、適切な SoR サービスへルーティングする。
- **責務**: 共通認証、レート制限、ログ記録、プロトコル変換。

### 3.2. サーキットブレーカー (Circuit Breaker)
- **内容**: 同期連携において、呼び出し先のサービスが連続してエラーを返した場合、通信を遮断してフォールバック処理を行う。
- **目的**: 障害がシステム全体に波及する「連鎖倒産」を防止する。

### 3.3. べき等性 (Idempotency) の確保
- **原則**: 同一のリクエストが複数回送信されても、システムの状態が一度しか変わらないことを保証する。
- **対策**: リクエストヘッダーに `Idempotency-Key`（UUID等）を含め、SoR 側で処理済みかどうかを判定する。

## 4. 📂 ファイル連携
大量データの受け渡しや、古い外部システムとの連携にはファイルベースの方式を採用する。

- **フォーマット**: CSV (UTF-8, ヘッダーあり) または JSON Lines。
- **転送プロトコル**: SFTP または オブジェクトストレージ（S3/Azure Blob等）。
- **排他制御**: 処理中フラグや完了ファイルの配置により、二重取り込みを防止する。

## 5. 🚨 エラー伝播とリトライ
- **リトライ方針**: 一時的なネットワークエラーに対しては、指数関数的バックオフ（Exponential Backoff）を用いた自動リトライを行う。
- **デッドレターキュー (DLQ)**: 非同期処理でリトライ上限に達したメッセージは、専用のキュー（隔離領域）に保存し、後で手動解析可能にする。
