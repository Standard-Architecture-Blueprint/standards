# 3. Design Guidelines (共通設計規約)

## 📌 概要
本ディレクトリは、Standard Architecture Blueprint (SAB) プロジェクトにおけるアーキテクチャの共通指針を定める。AP試験の「ソフトウェア設計」における「内部設計・外部設計」の標準化を行い、マイクロサービス間の疎結合性と、単体リポジトリ内の高凝集性を両立させることを目的とする。

## 🏛️ 管理資産（設計ガイドライン）

### [domain-modeling.md](./domain-modeling.md)
**ドメインモデリングと整合性方針**
- **境界付けられたコンテキスト**: 13のリポジトリが担当する業務範囲の定義。
- **データ整合性**: 強整合性（ACID特性）と結果整合性（BASE特性）の使い分け。
- **目的**: データベース設計における「正規化」と、マイクロサービス間での「データ重複」のトレードオフを統治する。

### [api-standard.md](./api-standard.md)
**インターフェース設計規約**
- **通信プロトコル**: RESTful API 規約（リソース定義、HTTPメソッド、ステータスコード）。
- **共通レスポンス**: `ApiResponse<T>` 構造の統一。
- **目的**: サービス間のインターフェース（API）を標準化し、相互運用性を高める。

### [architecture-patterns.md](./architecture-patterns.md)
**アーキテクチャ・パターンの分離**
- **レイヤード構造**: オニオンアーキテクチャまたはクリーンアーキテクチャの採用基準。
- **SoE/SoR/SoI**: システムの特性（記録重視か、エンゲージメント重視か）に応じた分離ルール。
- **目的**: 「関心の分離」を徹底し、保守性とテスト容易性を最大化する。

## 📐 設計の基本原則 (Design Principles)
AP試験の「ソフトウェア構造設計」の評価指標に基づき、以下の原則を遵守せよ。

1.  **疎結合・高凝集 (Loose Coupling, High Cohesion)**:
    - モジュール間の依存を最小限にし、特定の機能が単一のモジュールに集中していること。
2.  **インターフェースへの依存**:
    - 具体的な実装クラスではなく、インターフェース（抽象）に依存させることで、交換容易性を確保する（依存性注入：DI の前提）。
3.  **DRY (Don't Repeat Yourself)**:
    - 共通ロジックは `common-library` へ抽出し、13のリポジトリで重複させない。
4.  **フェイルセーフ / フォールトトレラント**:
    - 外部サービス呼び出しの失敗を想定し、システム全体が連鎖的に崩壊しない設計（サーキットブレーカー等）を考慮する。

## 📝 設計レビューの合否基準
設計成果物（Mermaid等）は、以下のチェックリストを満たす必要がある。
- [ ] 要件定義（RD）の要件IDと紐付いているか。
- [ ] 通信インターフェースは `api-standard.md` に準拠しているか。
- [ ] データの更新単位（トランザクション境界）が明確か。
- [ ] 例外ハンドリングの責任境界が定義されているか。
