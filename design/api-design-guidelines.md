# API Design Guidelines (Web API 標準規約)

## 1. 📌 概要
本ドキュメントは、システム間およびフロントエンド（SoE）とバックエンド（SoR）間の通信におけるインターフェース標準を定義する。REST (Representational State Transfer) の原則に基づき、一貫性と保守性の高い API 設計を目的とする。

## 2. 🛣️ エンドポイント設計
### 2.1. URL構成
- リソース名は **複数形の名詞** を使用する。
- 操作（動詞）をURLに含めず、HTTPメソッドで表現する。
- 小文字のケバブケース（kebab-case）を推奨する。

| 操作 | メソッド | パス | 意味 |
|:---|:---|:---|:---|
| 一覧取得 | GET | `/api/v1/users` | ユーザ一覧を取得する |
| 個別取得 | GET | `/api/v1/users/{id}` | 特定のユーザを取得する |
| 新規作成 | POST | `/api/v1/users` | 新しいユーザを作成する |
| 更新 | PUT | `/api/v1/users/{id}` | 既存ユーザを置換更新する |
| 部分更新 | PATCH | `/api/v1/users/{id}` | 既存ユーザの一部を更新する |
| 削除 | DELETE | `/api/v1/users/{id}` | ユーザを削除する |

### 2.2. バージョニング
- APIの破壊的変更を管理するため、パスにバージョンを含める。
- 例: `/api/v1/...`

## 3. 📡 リクエストとレスポンス
### 3.1. データフォーマット
- リクエスト・レスポンス共に **JSON** 形式を標準とする。
- プロパティ名は **キャメルケース（camelCase）** で統一する。

### 3.2. 共通レスポンス構造
クライアント側のエラーハンドリングを共通化するため、以下のエンベロープ形式を採用する。
```json
{
  "status": "success", 
  "data": {
    "id": 1,
    "name": "Tanaka Taro"
  },
  "error": null,
  "timestamp": "2026-05-03T16:00:00Z"
}
```

## 4. 🚦 HTTPステータスコード
応用情報技術者試験の「プロトコル」および「Webシステム」の範囲に基づき、セマンティクス（意味論）に忠実なコードを選択する。

| 分類 | コード | 意味 | ユースケース |
|:---|:---|:---|:---|
| **2xx (成功)** | 200 | OK | GETによる取得、PUT/PATCHによる更新成功時 |
| | 201 | Created | POSTによる新規リソース作成成功時 |
| | 204 | No Content | DELETE成功時など、レスポンスボディが不要な場合 |
| **4xx (クライアント誤差)** | 400 | Bad Request | 入力バリデーションエラー、リクエスト形式不備 |
| | 401 | Unauthorized | 未認証、または有効期限切れのトークンでのアクセス |
| | 403 | Forbidden | 認証済みだが、対象リソースへのアクセス権限がない場合 |
| | 404 | Not Found | URLが不正、または指定されたIDのリソースが存在しない |
| | 409 | Conflict | 楽観的ロックエラー、ユニーク制約違反など状態の競合 |
| **5xx (サーバー誤差)** | 500 | Internal Server Error | サーバー側での未捕捉の例外、致命的な不具合 |
| | 503 | Service Unavailable | メンテナンス中、または過負荷による一時的な受付不可 |

## 5. 🛡️ バリデーションとエラーハンドリング
### 5.1. バリデーションの責任分解
- **SoE（フロントエンド）**:
  - 役割: ユーザーへの即時フィードバック。
  - 内容: 必須チェック、桁数、正規表現（メールアドレス形式等）、型チェック。
- **SoR（バックエンド）**:
  - 役割: データの整合性保護（防御的プログラミング）。
  - 内容: ビジネスルール（残高不足、重複チェック）、権限チェック、DB整合性チェック。

### 5.2. エラーレスポンス形式
エラー発生時は `status: "error"` とし、以下の構造で詳細を返却する。

```json
{
  "status": "error",
  "data": null,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容に不備があります",
    "details": [
      {
        "field": "email",
        "issue": "format_invalid",
        "message": "メールアドレスの形式が正しくありません"
      }
    ]
  },
  "timestamp": "2026-05-03T16:05:00Z"
}
```

## 6. 🔍 検索・ページネーション・ソート
一覧取得API（`GET`）におけるデータ抽出条件は、すべてクエリパラメータで制御する。これにより、ブラウザのブックマークや「戻る」ボタンの挙動との親和性を確保する。

### 6.1. ページネーション (Pagination)
大量のデータを分割して取得する際は、以下のパラメータを使用する。
- `page`: 取得するページ番号（1オリジンを推奨）。
- `size`: 1ページあたりの項目数（デフォルト値と最大値をサーバー側で定義）。

**例**: `/api/v1/orders?page=2&size=50`

### 6.2. ソート (Sorting)
並び替え条件は、`プロパティ名,方向` の形式で指定する。
- 方向は `asc` (昇順) または `desc` (降順) とする。
- 複数の条件を指定する場合は、パラメータを複数繋げる。

**例**: `/api/v1/orders?sort=createdAt,desc&sort=id,asc`

### 6.3. フィルタリング (Filtering)
特定条件による絞り込みは、フィールド名をキーとして指定する。
- 部分一致や範囲指定が必要な場合は、フィールド名に接尾辞を付ける運用を検討する（例: `name_like`, `price_min`）。

**例**: `/api/v1/users?status=active&role=admin`

## 7. 🔒 セキュリティ (Security)
応用情報技術者試験の「情報セキュリティ」分野に基づき、APIレベルでの多層防御を定義する。

### 7.1. 認証トークンの伝送
- 全ての保護されたエンドポイントは、**JWT (JSON Web Token)** を要求する。
- トークンは HTTP ヘッダー `Authorization: Bearer <JWT>` に格納して送信する。

### 7.2. 機密情報の保護
- **メソッドの使い分け**: ログイン情報や個人情報などの機密データは、URL履歴やサーバーログに残ることを防ぐため、クエリパラメータ（GET）ではなく必ずリクエストボディ（POST）に含める。
- **HTTPSの強制**: 全ての通信は TLS (HTTPS) で暗号化し、中間者攻撃（MITM）を防止する。

### 7.3. 認可制御 (Access Control)
- **Role-Based Access Control (RBAC)**: トークンに含まれるロール（例: `ROLE_USER`, `ROLE_ADMIN`）に基づき、バックエンドのメソッドレベルで認可を行う。
- **CORS (Cross-Origin Resource Sharing)**: 許可されたオリジン（SoEのURL等）以外からのリクエストを Platform レイヤ（Gateway）で拒否する。

## 8. 🚀 パフォーマンスと制限
- **レート制限 (Throttling)**: 特定のクライアントからの過剰なアクセスによるリソース枯渇を防ぐため、単位時間あたりのリクエスト数制限を設ける。
- **Gzip圧縮**: レスポンスサイズが大きい場合は、HTTP圧縮を有効にして通信量を削減する。
